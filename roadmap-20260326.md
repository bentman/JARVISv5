# roadmap-20260326.md

## Purpose

Define the next voice implementation slice from current repository reality only.

This roadmap is intentionally limited to a **backend-first, STT-first** path that feeds microphone-assisted interaction into the **existing chat flow**. It does **not** include active TTS implementation work, and it does **not** assume any detached voice panel or stale upload path already exists in current source.

---

## Current Baseline

### Verified current state

- Backend source contains `POST /voice/transcribe` in `backend/api/main.py`.
- Backend source contains `ControllerService.transcribe()` in `backend/controller/controller_service.py`.
- Backend source contains `FasterWhisperSTTProvider` in `backend/voice/stt_provider.py`.
- Backend source contains `POST /voice/speak` and Piper provider wiring, but TTS catalog entries are disabled and Piper artifacts are not present at the expected runtime paths.
- `models/models.yaml` defines:
  - `whisper-stt` with `role="stt"`, `model_dir: models/voice/faster-whisper-base`, and `enabled: false`
  - `piper-tts` with `enabled: false`
  - `piper-tts-config` with `enabled: false`
- The repository contains `models/voice/faster-whisper-base/model.bin` and companion faster-whisper files.
- Current frontend source does **not** contain active voice UI or active source-level voice API client wiring.
- A searched built artifact references `/voice/transcribe/upload`, but current backend source does **not** implement that route. That built reference is stale and is not a source-of-truth input.

### Governing implementation decision

The near-term voice slice is:

1. **Speech-to-text only**
2. **Backend-first**
3. **Microphone input routes into the normal chat flow**
4. **No active TTS work until Piper model/config artifacts actually exist and are intentionally enabled**
5. **No detached voice panel assumption**

---

## Target Outcome for This Voice Slice

The finish line for this slice is:

- a user records speech from the existing chat surface,
- the frontend sends audio to a real backend STT upload endpoint,
- the backend transcribes with the existing controller/model-registry/provider path,
- the returned transcript is **automatically submitted** through the existing normal chat request path without an extra user send step,
- and the assistant response returns through the existing normal chat rendering path.

There is no TTS playback in this slice.

---

## Non-Goals

The following are explicitly out of scope for this roadmap:

- active TTS implementation
- Piper model download/provisioning work
- wake word support
- detached voice panel UI
- parallel voice conversation architecture separate from normal chat
- direct use of stale `/voice/transcribe/upload` behavior implied only by built frontend output

## Implementation Sequence

## Milestone 1 — Enable the Existing Backend STT Path

### Objective

Make the already-wired backend STT path actually selectable from the live model catalog.

### Required changes

- Update `models/models.yaml` so the existing `whisper-stt` entry becomes active.
- Keep the existing STT path backend-first:
  - `POST /voice/transcribe`
  - `ControllerService.transcribe()`
  - `ModelRegistry.select_model(... role="stt")`
  - `ModelRegistry.ensure_model_present(...)`
  - `FasterWhisperSTTProvider.transcribe_file(...)`
- Do **not** replace the current controller/model-registry/provider chain with direct endpoint-local model loading.

### Explicit decisions

- The STT model remains directory-based, using `model_dir: models/voice/faster-whisper-base`.
- The backend continues to instantiate `WhisperModel(self.model_dir)` through `FasterWhisperSTTProvider`.
- The existing JSON route `POST /voice/transcribe` remains valid for non-browser callers and testability.

### Acceptance criteria

- `POST /voice/transcribe` returns `200` with `{ transcript, model_id, profile, hardware }` when the STT model is selectable and transcription succeeds.
- `POST /voice/transcribe` remains fail-closed for empty input and runtime unavailability.
- The transcribe path still routes through `ControllerService.transcribe()`.
- No TTS behavior is changed.

---

## Milestone 2 — Add the Real Backend Upload Route for Browser Microphone Audio

### Objective

Introduce the missing backend upload route needed for browser-recorded audio while keeping transcription inside the existing STT pipeline.

### Required changes

- Add a new backend route: `POST /voice/transcribe/upload`.
- Implement it in `backend/api/main.py`.
- Add or extend schema support in `backend/api/schemas.py` only if needed for the response contract.
- Add focused backend tests for upload success and temp-file cleanup.

### Route contract

- Request type: `multipart/form-data`
- Field name: `audio`
- Input source: browser-recorded audio blob
- Response shape: exactly the same successful payload shape as `POST /voice/transcribe`
  - `transcript`
  - `model_id`
  - `profile`
  - `hardware`

### Execution contract

- The upload route writes the incoming audio to a bounded temporary file under the normal runtime data area.
- The route then calls `ControllerService.transcribe(temp_audio_path)`.
- The temporary file is deleted after success or failure.
- The upload route does **not** bypass the controller and does **not** instantiate `FasterWhisperSTTProvider` directly in the endpoint.

### Error contract

- Empty or missing upload fails closed.
- STT model unavailability returns the same availability-style failure class as the existing transcription path.
- Temp-file cleanup happens on both success and failure.

### Acceptance criteria

- A browser-compatible audio upload can be transcribed without requiring a filesystem path from the client.
- Temp files do not remain after request completion.
- The route reuses the existing backend STT chain rather than introducing parallel inference code.

---

## Milestone 3 — Integrate Microphone Capture Into the Existing Chat Surface

### Objective

Add microphone-assisted input to the current chat UI without creating a detached voice experience.

### Required frontend shape

- Reuse the existing chat surface.
- Do **not** introduce a new voice panel.
- Add microphone controls adjacent to the current message composer in the existing chat UI.

### Required frontend behavior

- Press/hold or explicit start/stop recording captures audio from the browser microphone.
- On recording completion, the frontend sends the audio blob to `POST /voice/transcribe/upload`.
- On successful transcription, the frontend immediately and automatically submits the returned transcript through the existing normal chat request flow.
- The normal chat request flow for this slice is the existing streaming task path, not a new voice-specific chat endpoint.
- The transcript is not staged in the composer for user review before send in this slice.

### Explicit chat-path decision

- After STT success, the transcript is auto-submitted through the existing task submission path used by the chat experience.
- The post-STT handoff behavior is fixed for this slice: **transcribe, then auto-send**.
- There is no intermediate composer-review step between STT success and chat submission.
- Assistant output continues to come from the existing chat response pipeline.
- No separate "voice conversation" backend route is introduced in this slice.

### Required file targets

- `frontend/src/api/taskClient.js`
- `frontend/src/App.jsx`
- `frontend/src/state/useChatState.js` if state coordination is needed

### UX requirements

- The user can see when recording is active.
- The user can see when transcription is in progress.
- After successful transcription, the UI clearly shows that the recognized transcript has been sent as the user's chat message.
- If transcription fails, the failure is shown inline in the existing chat experience.
- The transcript used for submission is visible as the normal user message content after auto-submit.

### Acceptance criteria

- A user can speak from the existing chat surface and receive an assistant response without typing.
- The assistant response still arrives through the same normal chat rendering flow already used for typed input.
- No detached voice-only UI is introduced.

---

## Milestone 4 — Harden the STT-to-Chat Path

### Objective

Make the STT-assisted chat path stable, fail-closed, and internally consistent.

### Required backend checks

- Unsupported or malformed browser audio uploads fail predictably.
- Transcription-empty cases fail predictably.
- Availability failures remain explicit and do not silently degrade into fake success.

### Required frontend checks

- Recording state resets correctly after success, cancellation, and failure.
- Duplicate submissions are prevented while transcription is in flight.
- Chat state remains consistent with existing task continuation behavior.

### Acceptance criteria

- Browser microphone interaction does not require any detached state machine outside the current chat flow.
- Failure cases are user-visible and do not leave stuck recording or submission state.
- The path is still STT-first only.

---

## Deferred Until Real Piper Availability Exists

TTS remains deferred until all of the following are true:

1. the Piper model file exists at the intended runtime path
2. the Piper config file exists at the intended runtime path
3. the corresponding catalog entries are intentionally enabled
4. runtime validation proves the backend can synthesize audio successfully

Until those conditions are met:

- `POST /voice/speak` is not part of the active delivery target
- no frontend playback controls should be added for TTS
- no roadmap milestone should claim end-to-end voice output capability

---

## Final Definition of Done

This voice slice is complete when all of the following are true:

- `whisper-stt` is actively selectable from the live catalog
- `POST /voice/transcribe/upload` exists and routes through `ControllerService.transcribe()`
- the existing chat UI can record microphone audio
- the recorded audio is transcribed by the backend STT path
- the resulting transcript is automatically submitted through the normal chat task path
- the transcript appears as the user's normal chat message after auto-submit
- assistant output appears in the normal chat interface
- no TTS implementation is included in the delivered slice

That is the full near-term voice slice.
