# roadmap-20260318.md

## Milestone M17: Privacy Controls Settings Surface
**Date:** 2026-03-18
**Status:** Draft — Pending Approval

---

### Goal

M17 closes the last named gap in the settings surface: `REDACT_PII_QUERIES` and `REDACT_PII_RESULTS` are typed in `Settings` and projected by `SafeConfigProjection`, but neither is writable via `POST /settings` and neither has a UI control. More critically, **neither flag gates any runtime behavior today** — both are inert. M17 wires both flags into the execution path and exposes them as user-controllable toggles.

`Project.md` §3.5 and §8.1 name privacy controls as first-class settings requirements alongside hardware profiles and budget governance.

**What is already implemented (do not re-implement):**

The `PrivacyExternalCallWrapper` in `backend/security/privacy_wrapper.py` and `backend/tools/executor.py` already provide comprehensive PII scanning on every tool call:
- `scan_tool_input()` — scans and redacts tool call inputs before execution
- `scan_tool_output()` — scans and redacts tool call results after execution
- `evaluate_and_prepare_external_call()` — gates all EXTERNAL-tier tool calls; deny-by-default; redacts payloads before external dispatch
- All events are logged via `SecurityAuditLogger` to `data/logs/security_audit.jsonl` (`PII_DETECTED`, `PII_REDACTED`, `EXTERNAL_CALL_INITIATED`, `PERMISSION_DENIED`)
- `PIIRedactor` supports `partial` mode (email, phone) and `strict` mode (adds SSN, credit card, IBAN, IP address); seven PII types total

This infrastructure is verified and complete (M5, M8). M17 does **not** touch any of it.

**What M17 adds:**

- **`REDACT_PII_QUERIES`** — gates redaction of the user input before it is injected into the LLM prompt context. Currently the flag exists but `LLMWorkerNode` injects raw `user_input` into the prompt unconditionally. When true, `user_input` in context should be passed through `PIIRedactor` in `strict` mode before prompt assembly.
- **`REDACT_PII_RESULTS`** — gates redaction of the final `llm_output` after validation, before commit and archive. Currently the flag exists but the controller never reads it. When true, `context["llm_output"]` is passed through `PIIRedactor` in `strict` mode after `ValidatorNode` confirms `is_valid=True`, before the COMMIT transition.
- **Settings write path** — both fields added to `EDITABLE_SETTINGS_ENV_KEYS` and `SettingsUpdateRequest` so they can be toggled at runtime via `POST /settings`.
- **UI controls** — two checkboxes in a Privacy subsection of `SettingsPanel.jsx`.
- **`.env.example` / `.env`** — both keys are currently absent from `.env.example`; they must be added.

**Design constraints:**
- Both flags are hot-applied (no restart required).
- `REDACT_PII_QUERIES` redaction is applied in `LLMWorkerNode` or immediately before prompt assembly in the controller, before the LLM call. The redacted text is used for the prompt only; `context["user_input"]` is preserved unredacted for episodic trace integrity.
- `REDACT_PII_RESULTS` redaction is applied in the controller after the VALIDATE phase passes, before the COMMIT transition. `context["result_pii_redacted"]` records whether redaction fired.
- Redaction mode is `strict` for both paths, consistent with the existing escalation-path redaction.
- The existing `PrivacyExternalCallWrapper` tool-side redaction is unaffected and unconditional — it does not need to be gated by these flags.

**Verified starting state (from `SYSTEM_INVENTORY.md` M16.2 entry and `CHANGE_LOG.md` 2026-03-18 08:27):**
- `backend/config/settings.py` — `REDACT_PII_QUERIES: bool = True`, `REDACT_PII_RESULTS: bool = False` defined in `Settings` and `SafeConfigProjection`; **neither** in `EDITABLE_SETTINGS_ENV_KEYS`

## T17.1 — Config + Env: Add Both Fields to Editable Settings

### Why

Both fields are absent from `EDITABLE_SETTINGS_ENV_KEYS`. Without this, `persist_settings_updates()` raises `unsupported editable setting` for both, and the `POST /settings` write path cannot accept either value. Both are boolean fields — the same serialization pattern as `allow_external_search` and `cache_enabled`. Both are also absent from `.env.example`, which is the canonical configuration template.

### Touch points

- `backend/config/settings.py`:
  - Add to `EDITABLE_SETTINGS_ENV_KEYS`:
    ```python
    "redact_pii_queries": "REDACT_PII_QUERIES",
    "redact_pii_results": "REDACT_PII_RESULTS",
    ```
  - Extend the boolean branch in `serialize_editable_setting_value()` to include both new field names alongside the existing boolean fields
  - `settings_update_restart_semantics()` requires no change — both fields are hot-applied by the existing logic that covers all non-`hardware_profile` fields
- `.env.example` — add under a `# Privacy / Redaction` comment:
  ```
  REDACT_PII_QUERIES=true
  REDACT_PII_RESULTS=false
  ```
- `.env` — add same two keys with same safe defaults

### Acceptance

- `serialize_editable_setting_value("redact_pii_queries", False)` returns `"false"` without raising.
- `serialize_editable_setting_value("redact_pii_results", True)` returns `"true"` without raising.
- `persist_settings_updates({"redact_pii_queries": False})` completes without error.
- `.env.example` and `.env` contain both keys with correct defaults.

## T17.2 — API: Settings Write Path for Both Privacy Fields

`SettingsUpdateRequest` uses `model_config = ConfigDict(extra="forbid")` — undeclared fields are rejected with 422. Both fields must be declared before `POST /settings` can accept writes to them. No custom validators are needed; Pydantic handles `bool | None` natively. `GET /settings` already returns both fields via `SettingsResponse` — no change required there.

- `backend/api/main.py` — no structural changes required; both fields flow through `get_safe_config_projection()` which already projects them; verify the `POST /settings` handler correctly passes both through after T17.1

- `POST /settings {"redact_pii_results": true}` returns 200 with updated value.

## T17.3 — Workflow: Wire `REDACT_PII_QUERIES` into LLM Prompt Path

### Why

`LLMWorkerNode.execute()` assembles the prompt from `context["user_input"]` and `context["messages"]` with no redaction gate. When `REDACT_PII_QUERIES=True`, any PII in the user's query is forwarded verbatim into the LLM prompt. The flag should gate a `strict`-mode redaction pass on the text that enters the prompt.

**Implementation approach:** `LLMWorkerNode` does not currently read `Settings`. The cleanest insertion point is in the controller, before `LLMWorkerNode` is called — the controller already reads `Settings()` at the top of `run()`. A redacted copy of `user_input` is stored in context under a dedicated key (e.g. `context["llm_prompt_input"]`), and `LLMWorkerNode` is updated to prefer `llm_prompt_input` over `user_input` when building the prompt. This preserves the original `user_input` in the working state and episodic trace unchanged.

**Alternatively**, if the agent prefers a simpler approach: apply the redaction directly in the controller before the EXECUTE phase sets off the DAG, overwriting `context["user_input"]` only within the prompt assembly scope. Either approach is acceptable; the key constraint is that the episodic trace must record the original user input, not the redacted version.

### Touch points

- `backend/controller/controller_service.py`:
  - After the PLAN phase and before the EXECUTE phase DAG runs, when `settings.REDACT_PII_QUERIES` is True:
    - Apply `create_default_redactor().redact(user_input, mode="strict")`
    - Store result in `context["llm_prompt_input"]`
    - Set `context["query_pii_redacted"] = redaction_result.pii_detected`
  - When False: set `context["llm_prompt_input"] = context["user_input"]` and `context["query_pii_redacted"] = False`
- `backend/workflow/nodes/llm_worker_node.py`:
  - In `execute()`, prefer `context.get("llm_prompt_input") or context.get("user_input", "")` as the source for prompt user input — this applies to **both** the standalone prompt path (no prior messages) and the `messages`-present path where prior conversation turns are injected. The redacted prompt input must replace the raw `user_input` reference in both branches of prompt assembly.
  - No other changes to `LLMWorkerNode`

### Acceptance

- When `REDACT_PII_QUERIES=True` and `user_input` contains a detectable PII pattern, `context["llm_prompt_input"]` contains the redacted form and `context["query_pii_redacted"] == True`.
- When `REDACT_PII_QUERIES=True` and `user_input` contains no PII, `context["llm_prompt_input"] == context["user_input"]` and `context["query_pii_redacted"] == False`.
- When `REDACT_PII_QUERIES=False`, `context["llm_prompt_input"] == context["user_input"]` and `context["query_pii_redacted"] == False`.
- `context["user_input"]` is never modified — episodic trace preserves original.
- Existing task execution paths are unaffected when `REDACT_PII_QUERIES=False` (the default).

### Evidence

- `backend/.venv/Scripts/python scripts/validate_backend.py --scope unit` — PASS (no regressions)

## T17.4 — Controller: Wire `REDACT_PII_RESULTS` into Post-Validation Result Path

### Why

`REDACT_PII_RESULTS` is never read in the controller today. When enabled, the final `llm_output` should be passed through `PIIRedactor` in `strict` mode after `ValidatorNode` confirms `is_valid=True` and before the COMMIT transition. This prevents PII generated by the model from being persisted in the working state archive or returned in the API response.

The redaction must be applied at the right point in the commit sequence. The controller currently persists the assistant message to the working-state transcript (via `append_task_message`) and writes to semantic memory before the COMMIT transition. Both of these writes must happen **after** result redaction — otherwise the unredacted output is persisted to task history and the semantic store regardless of the flag. The redaction block must therefore precede any assistant-message persistence and any semantic memory write on the success path, as well as preceding the COMMIT transition.

### Touch points

- `backend/controller/controller_service.py`:
  - After the VALIDATE phase passes (`context["is_valid"] == True`) and before the `fsm.transition(ControllerState.COMMIT)` call:
    ```python
    if bool(settings.REDACT_PII_RESULTS):
        _redactor = create_default_redactor()
        _raw_output = str(context.get("llm_output", ""))
        _result_redaction = _redactor.redact(_raw_output, mode="strict")
        context["llm_output"] = _result_redaction.redacted
        context["result_pii_redacted"] = _result_redaction.pii_detected
    else:
        context["result_pii_redacted"] = False
    ```
  - `create_default_redactor` is already imported (`from backend.security.redactor import create_default_redactor`) — no new import needed
  - The redaction block must execute before: (1) any `append_task_message` call that persists the assistant turn, (2) any `store_knowledge` / semantic memory write for the output, and (3) the `fsm.transition(ControllerState.COMMIT)` call. Confirm the actual call order in the current controller and insert accordingly.

### Acceptance

- When `REDACT_PII_RESULTS=True` and `llm_output` contains a PII pattern, the committed output has redaction tokens; `context["result_pii_redacted"] == True`.
- When `REDACT_PII_RESULTS=True` and `llm_output` contains no PII, output is unchanged; `context["result_pii_redacted"] == False`.
- When `REDACT_PII_RESULTS=False` (default), output is unchanged; `context["result_pii_redacted"] == False`.
- Redaction is not applied on the failed/invalid path.
- Escalation-path outputs (Ollama and cloud) are subject to the same gate — they set `llm_output` in context before the VALIDATE phase, so the same post-validation check applies.

- `backend/.venv/Scripts/python scripts/validate_backend.py --scope unit` — PASS (no regressions)

## T17.5 — Frontend: Privacy Control Toggles in Settings Panel

### Why

With T17.1 and T17.2 complete, both fields are writable via `POST /settings`. The UI currently shows them only in the read-only "Settings (Current Snapshot)" section. Two checkboxes in a dedicated **Privacy** subsection give the user runtime control without requiring `.env` edits.

### Touch points

- `frontend/src/components/SettingsPanel.jsx`:
  - Add `"redact_pii_queries"` and `"redact_pii_results"` to `EDITABLE_FIELDS` array
  - Extend `pickEditableSettings()` with safe defaults:
    ```javascript
    redact_pii_queries: Boolean(settings?.redact_pii_queries ?? true),
    redact_pii_results: Boolean(settings?.redact_pii_results ?? false),
    ```
  - `editableSettingsEqual()` requires no change — `EDITABLE_FIELDS.every(...)` covers new fields automatically
  - Render a **Privacy** subsection between the Cache section and the Model Escalation section:
    - `Redact PII in Queries` checkbox → `redact_pii_queries`
    - `Redact PII in Results` checkbox → `redact_pii_results`
    - Read-only note: `"Strict mode: email, phone, SSN, credit card, IBAN, IP address"`
  - Follow existing panel styling — same checkbox/label pattern as `Allow External Search` and `Cache Enabled`
  - Existing Save / Cancel button pair covers the new fields — no additional save button

### Acceptance

- Both checkboxes render with correct current values from `GET /settings`.
- Toggling either marks the panel dirty and enables Save.
- Save calls `POST /settings` with the changed field(s); panel updates on success.
