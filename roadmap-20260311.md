## Milestone M14: `SearchWebNode` + Search Provider Controls
**Date:** 2026-03-11
**Status:** Draft — Pending Approval

---

### Goal

M14 closes two related gaps: (1) the absence of a dedicated `search_web` workflow node as required by `Project.md` §5.1, and (2) the complete disconnect between implemented search provider controls and runtime behavior. Three providers are operational at the library layer (SearXNG, DuckDuckGo, Tavily) but no settings field, policy gate, or provider tier distinction has any effect on execution. Research intent routes through the generic `ToolCallNode` with hardcoded `allow_external: True`. This milestone builds the proper node, wires the controls, and establishes the local → free-external → paid-escalation tier model.

**Verified starting state (from `SYSTEM_INVENTORY.md`, M13 entry):**
- `backend/search/providers/`: SearXNG, DuckDuckGo, Tavily providers implemented and registered in `ProviderLadder`
- `backend/search/policy.py`: `decide_external_search()` exists; binary allow/deny + budget check only
- `backend/tools/search_tools.py`: `build_search_tool_dispatch_map()` and `SearchWebInput` exist; `allow_external` baked in at construction time
- `backend/workflow/nodes/tool_call_node.py`: dispatches `search_web` generically; no `SearchWebNode` exists
- `backend/config/settings.py`: `ALLOW_EXTERNAL_SEARCH` and `DEFAULT_SEARCH_PROVIDER` fields exist and are exposed via `/settings` API but are not read at runtime
- `backend/controller/controller_service.py`: research intent auto-injects `tool_call` with hardcoded `allow_external: True`; `ToolCallNode` present in `node_registry`; no `search_web` node registered

## T14.1 — Provider Tier Classification

### Why

`backend/search/providers/base.py` — `SearchProviderBase` has no tier attributes. The ladder treats all three providers identically: try in order, fall through on failure. SearXNG, DDG, and Tavily have fundamentally different cost and privacy profiles that require explicit policy separation:

| Provider | Traffic | Cost | Gate |
|----------|---------|------|------|
| SearXNG | Local Docker only | $0 | Always allowed |
| DuckDuckGo | External HTTP | $0 | Requires `ALLOW_EXTERNAL_SEARCH` |
| Tavily | External API | Paid | Requires `ALLOW_EXTERNAL_SEARCH` + `ALLOW_PAID_SEARCH` + key + budget |

Without declared tier attributes, the policy layer (T14.3) and `SearchWebNode` (T14.4) cannot make tier-aware decisions without hardcoding provider names.

### Touch points

- `backend/search/providers/base.py` — add `is_external: bool = False` and `is_paid: bool = False` class attributes to `SearchProviderBase`
- `backend/search/providers/searxng.py` — `is_external = False`, `is_paid = False`
- `backend/search/providers/ddg.py` — `is_external = True`, `is_paid = False`
- `backend/search/providers/tavily.py` — `is_external = True`, `is_paid = True`

### Acceptance

- `SearchProviderBase` declares `is_external` and `is_paid` with correct defaults.
- All three provider classes declare their tier values explicitly.
- Existing `ProviderLadder` behavior is unchanged — attributes are additive only.
- No existing tests broken.

### Evidence

- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS (no regressions)

## T14.2 — Settings Wiring for Search Fields

### Why

`backend/config/settings.py` — `ALLOW_EXTERNAL_SEARCH` and `DEFAULT_SEARCH_PROVIDER` are present but ignored at runtime. `SEARCH_SEARXNG_URL` and `TAVILY_API_KEY` are read via raw `os.getenv()` inside provider files — they are invisible to the Settings API, the `/settings` endpoint, and any operator tooling. `ALLOW_PAID_SEARCH` does not exist at all.

This task makes all search-relevant configuration flow through `Settings()` consistently, so T14.3 and T14.4 can read a single source of truth.

### Touch points

- `backend/config/settings.py` — add fields:
  - `SEARCH_SEARXNG_URL: str = "http://searxng:8080/search"` (move from `os.getenv` in `searxng.py`)
  - `TAVILY_API_KEY: str = ""` (move from `os.getenv` in `tavily.py`)
  - `ALLOW_PAID_SEARCH: bool = False` (new; gates Tavily specifically)
  - `ALLOW_EXTERNAL_SEARCH` — already present; no schema change, wire to runtime in T14.3/T14.4
  - `DEFAULT_SEARCH_PROVIDER` — already present; wire to ladder `preferred_provider` in T14.4
- `backend/search/providers/searxng.py` — remove `os.getenv("SEARCH_SEARXNG_URL", ...)` call; accept `base_url` as constructor parameter with default from `Settings()`
- `backend/search/providers/tavily.py` — remove `os.getenv("TAVILY_API_KEY", ...)` call; accept `api_key` as constructor parameter with default from `Settings()`
- `backend/api/main.py` — expose `SEARCH_SEARXNG_URL`, `TAVILY_API_KEY` (key presence as bool only — do not expose the key value), and `ALLOW_PAID_SEARCH` in `get_safe_config_projection()` and `SettingsResponse` schema
- `backend/api/schemas.py` — extend `SettingsResponse` and `SettingsUpdateRequest` as needed
- `.env.example` — add `ALLOW_PAID_SEARCH=false`

### Acceptance

- `Settings()` instance exposes all five search fields.
- `GET /settings` response includes `allow_external_search`, `allow_paid_search`, `default_search_provider`, `searxng_url`, and `tavily_key_configured` (bool — not the key value).
- `SearXNGProvider` and `TavilyProvider` no longer call `os.getenv()` directly.
- Existing settings tests pass; new field projections covered.

## T14.3 — Search Policy Three-Path Tier Enforcement

### Why

`backend/search/policy.py` — `decide_external_search()` is binary: `allow_external: bool` + budget check. It cannot distinguish between free-external (DDG) and paid-external (Tavily). The existing function signature will be preserved for backward compatibility with current callers; a new `decide_search_tier()` function will be added alongside it.

The three-path logic:
1. **Local tier (SearXNG):** `is_external=False` — always allowed; no policy check required.
2. **Free-external tier (DDG):** `is_external=True`, `is_paid=False` — requires `ALLOW_EXTERNAL_SEARCH=true`; no budget check.
3. **Paid-external tier (Tavily):** `is_external=True`, `is_paid=True` — requires `ALLOW_EXTERNAL_SEARCH=true` + `ALLOW_PAID_SEARCH=true` + `TAVILY_API_KEY` non-empty + budget check.

### Touch points

- `backend/search/policy.py` — add `decide_search_tier(provider, settings, budget_ledger, budget_config) -> tuple[bool, dict]`; returns explicit decision with `tier`, `code`, and `reason` in the result dict; existing `decide_external_search()` unchanged
- `tests/unit/test_search_policy.py` — add tier-path cases: local-always-allowed, free-external-denied, free-external-allowed, paid-denied-no-key, paid-denied-no-flag, paid-denied-budget, paid-allowed

### Acceptance

- `decide_search_tier()` returns correct allow/deny for each of the seven cases above.
- SearXNG always returns allow regardless of settings.
- DDG denied when `ALLOW_EXTERNAL_SEARCH=false`.
- Tavily denied when key absent even if both flags are true.
- Tavily denied when budget exceeded.
- Existing `decide_external_search()` behavior unchanged and existing tests still pass.

### Evidence

- `backend/.venv/Scripts/python -m pytest tests/unit/test_search_policy.py -q` — PASS
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS (no regressions)

## T14.4 — `SearchWebNode` Implementation

### Why

`Project.md` §5.1 names `search_web` as a first-class node type. Currently no such node exists — research queries are dispatched through `ToolCallNode` with a generic `tool_call` dict constructed in the controller. `SearchWebNode` should own all search-specific concerns: reading settings, applying tier policy, invoking the ladder with the correct provider subset, and injecting structured results into context.

### Touch points

- `backend/workflow/nodes/search_web_node.py` — new file; `SearchWebNode(BaseNode)` with `execute(context)`:
  - Reads `Settings()` to get `ALLOW_EXTERNAL_SEARCH`, `ALLOW_PAID_SEARCH`, `TAVILY_API_KEY`, `DEFAULT_SEARCH_PROVIDER`, `SEARCH_SEARXNG_URL`
  - Builds `ProviderLadder` with only tier-permitted providers (always include SearXNG; include DDG if `ALLOW_EXTERNAL_SEARCH`; include Tavily if `ALLOW_EXTERNAL_SEARCH` + `ALLOW_PAID_SEARCH` + key present)
  - Applies `decide_search_tier()` per active provider; falls through on deny to next tier
  - Passes `DEFAULT_SEARCH_PROVIDER` as `preferred_provider` to `ProviderLadder.search()`
  - Injects into context: `search_results` (list of `SearchResultItem` dicts), `search_provider` (name of successful provider), `search_ok` (bool), `search_error` (reason string on failure)
  - Does not call `ToolCallNode` — standalone execution
- `backend/workflow/__init__.py` — export `SearchWebNode`
- `backend/workflow/nodes/__init__.py` (if exists) — add `SearchWebNode` import
- `tests/unit/test_nodes.py` or `tests/unit/test_search_web_node.py` — unit tests with stubbed ladder

### Acceptance

- `SearchWebNode` is importable from `backend.workflow`.
- With `ALLOW_EXTERNAL_SEARCH=false`: only SearXNG is attempted; DDG and Tavily are excluded from the ladder instance.
- With `ALLOW_EXTERNAL_SEARCH=true`, `ALLOW_PAID_SEARCH=false`: SearXNG + DDG attempted; Tavily excluded.
- With all flags true and key present: all three providers in ladder; `DEFAULT_SEARCH_PROVIDER` respected as preferred.
- `search_results` in context is a list of dicts with `title`, `url`, `snippet`, `source_provider`.
- `search_ok=False` with `search_error` populated when all providers fail.
- Deterministic: same settings + same stub provider response → same context output.

### Evidence

- `backend/.venv/Scripts/python -m pytest tests/unit/test_nodes.py -q` — PASS (or dedicated test file)
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS (no regressions)

## T14.5 — Controller Research Graph Cleanup

### Why

`backend/controller/controller_service.py` — research intent auto-injection constructs and injects a `tool_call` dict with hardcoded `allow_external: True`, then `_with_tool_call_node()` inserts `ToolCallNode` into the graph. With `SearchWebNode` now available and registered, research intent should route to it directly. The generic injection block and the `_with_tool_call_node()` helper should no longer be triggered by research intent — they remain for explicit external `tool_call` requests (file tools, fetch_url, etc.).

`SearchWebNode` is added to `node_registry`. The research-intent graph is: `router → context_builder → search_web → llm_worker → validator`.

### Touch points

- `backend/controller/controller_service.py`:
  - Add `"search_web": SearchWebNode()` to `node_registry`
  - Add `"search_web"` to `phase_to_nodes[ControllerState.EXECUTE]`
  - Replace the research auto-injection block (currently builds and injects a `tool_call` dict) with a direct graph selection: when `intent == "research"`, compile a research graph that includes `search_web` between `context_builder` and `llm_worker`
  - `_with_tool_call_node()` helper is unchanged — still used for explicit `tool_call` in context (non-research paths)
- `backend/workflow/plan_compiler.py` — optionally extend `compile_plan_to_workflow_graph()` to accept intent and return the appropriate graph variant; or keep graph selection in the controller — document the chosen approach

### Acceptance

- Research intent execution order: `["router", "context_builder", "search_web", "llm_worker", "validator"]`
- Chat and code intent execution order unchanged: `["router", "context_builder", "llm_worker", "validator"]`
- Explicit `tool_call` in context (non-research) still routes through `ToolCallNode` via `_with_tool_call_node()`.
- `context["tool_call"]` is not injected for research intent (no longer needed).
- All three intent paths covered by integration tests.

### Evidence

- `backend/.venv/Scripts/python -m pytest tests/unit/test_controller_service_integration.py -q` — PASS
