## Milestone M12: Memory-to-Prompt Wiring, Intent Routing Depth & Repo Alignment
**Date:** 2026-03-06
**Status:** Draft — Pending Approval

---

### Goal

M11 delivered the execution pipeline, memory infrastructure, search, and UI scaffolding.
M12 makes those capabilities actually work together as a coherent daily-use assistant.

The most critical finding from code review: **the LLM prompt ignores the assembled context.**
`ContextBuilderNode` retrieves conversation history, attachment context, and semantic memory —
then writes it to `context["messages"]`. `LLMWorkerNode` reads `context["user_input"]` and
builds a flat prompt that never consults `context["messages"]`. Every prior-turn conversation,
every retrieved memory artifact, every file attachment is silently discarded at inference time.

Until T12.1 is resolved, multi-turn continuity, memory recall, and file context are inoperable
at the LLM level regardless of what the UI or storage layers do.

## T12.1 — Wire Context Messages into LLM Prompt

### Why
`LLMWorkerNode.execute()` builds its prompt solely from `context["user_input"]`. It never
reads `context["messages"]`, which `ContextBuilderNode` populates with conversation history,
attachment context, and retrieved semantic memory. This means:
- Multi-turn conversations have no history at inference time
- File uploads are injected into messages but the LLM never sees them
- Semantic memory retrieval runs and is immediately discarded

`Project.md` §5.2 step 3 is explicit: "context_builder retrieves relevant memory" feeds into
step 4 "workflow DAG executes nodes." The context chain is broken between these two nodes.

### What's needed
`LLMWorkerNode` needs to read `context["messages"]` and incorporate the message history into
the prompt it sends to `llama_cpp`. The existing `messages` list is already formatted as
`[{"role": "system"|"user"|"assistant", "content": "..."}]` — this maps directly to a
chat-style prompt structure.

The fix must remain deterministic and reproducible. Seed, stop sequences, and output
normalization behavior are unchanged. The existing single-turn prompt is the fallback when
`context["messages"]` is absent or empty.

**Touch points:** `backend/workflow/nodes/llm_worker_node.py`,
`tests/unit/test_nodes.py` or a dedicated LLM worker test.

### Acceptance
- When `context["messages"]` is populated, the LLM prompt includes prior turns and system
  context in dependency order.
- Single-turn behavior (no messages) is unchanged.
- Output normalization, stop sequences, and seed behavior are preserved.
- Deterministic replay: same messages + same seed produces same output.
- Unit tests cover multi-turn prompt construction and single-turn fallback.

### Evidence
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS excerpt

## T12.2 — Research Intent Routing

### Why
`RouterNode` is four lines: `"code"` if "code" is in the input, otherwise `"chat"`. `Project.md`
§5.1 defines three intent paths: `chat`, `code`, `research`. §2 lists "search, research
synthesis" as primary daily-use paths. The research path — which should route toward the
`tool_call`/search pipeline rather than direct LLM generation — doesn't exist.

With T12.1 done, the infrastructure to differentiate these paths is in place. This task adds
a deterministic research classifier and connects `intent = research` to the existing search
tooling.

### What's needed
A rule-based research intent classifier in `RouterNode` (heuristics only — deterministic
control invariant, `Project.md` §1). When `intent = research`, the workflow compiler or
controller should direct execution toward the search/tool path rather than straight to
`llm_worker`. The `tool_call` node and all three search providers are already present and
operational — this task connects routing intent to that infrastructure.

**Touch points:** `backend/workflow/nodes/router_node.py`,
`backend/workflow/plan_compiler.py`,
`backend/controller/controller_service.py`,
`tests/unit/test_nodes.py` or equivalent.

### Acceptance
- `intent = research` is produced for inputs expressing lookup, search, or research intent.
- `intent = code` and `intent = chat` are unchanged.
- Research-intent tasks route toward the search/tool execution path.
- Classification is deterministic: same input → same intent on every run.
- All three intent paths are covered by unit tests.

### Evidence
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS excerpt

## T12.3 — Expose Memory Query API (`/memory/search`)

### Why
`EpisodicMemory.search_decisions()`, `EpisodicMemory.search_tool_calls()`, and
`MemoryManager.retrieve_knowledge()` are all implemented and tested. None of them are
reachable via any API endpoint. From the UI's perspective, memory is write-only.

`Project.md` §3.5 requires "memory/search controls" in the web client. §11.2 names it as
an interface element. §2 lists "knowledge recall" as a daily-use path. Without an API
surface, none of that is reachable.

### What's needed
A `GET /memory/search` endpoint that accepts a query string and returns results from both
semantic memory (`retrieve_knowledge`) and episodic trace (`search_decisions`). This is
additive — no changes to existing endpoints. The response shape should be consistent with
what a frontend memory panel would consume.

The endpoint should be covered by a unit test consistent with existing API test patterns
in `tests/unit/`.

**Touch points:** `backend/api/main.py`, `backend/api/schemas.py`,
new or updated test in `tests/unit/`.

### Acceptance
- `GET /memory/search?q=<query>` returns semantic and episodic results.
- Response includes source type, content, and timestamp for each result.
- Empty query returns a clear validation error, not a 500.
- Existing endpoints are unaffected.
- Unit test covers query, empty-query, and zero-results cases.

### Evidence
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS excerpt

## T12.5 — Frontend Structure: Extract `state/`, `styles/`, `utils/`

### Why
`Project.md` §13 defines `frontend/src/state/`, `styles/`, and `utils/` as distinct modules.
`App.jsx` is 687 lines with chat state logic, render utilities, and style constants
co-located. This is a pure refactor — no logic changes, no new dependencies, no behavioral
change. It aligns the frontend layout with the architecture spec and reduces the maintenance
surface of `App.jsx` as features are added.

Three modules extracted from existing code:
- `state/useChatState.js` — chat state, refs, and handlers as a custom React hook
- `styles/theme.js` — color/spacing constants from `Project.md` §11.1
- `utils/renderHelpers.jsx` — the four render/utility functions at the top of `App.jsx`

`App.jsx` becomes a thin composition shell. `SettingsPanel.jsx` and `WorkflowVisualizer.jsx`
are not touched.

**Touch points:** `frontend/src/App.jsx`, new files under `frontend/src/state/`,
`frontend/src/styles/`, `frontend/src/utils/`.

### Acceptance
- All three new modules exist with content.
- `App.jsx` ≤ 150 lines.
- `SettingsPanel.jsx` and `WorkflowVisualizer.jsx` unmodified.
- `npm --prefix frontend run build` passes clean.
- No behavioral or visual change.

### Evidence
- File existence, `App.jsx` line count, `npm --prefix frontend run build` PASS excerpt.
