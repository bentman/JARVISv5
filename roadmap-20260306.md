## Task T12.1 — Create `data/retrieval/` Runtime Directory

### Why
`Project.md` §13 defines `data/retrieval/` as a canonical runtime artifact path. The directory
does not exist. Retrieval artifacts currently have no canonical home, which creates inconsistency
between the architecture spec and the repo layout.

### Current state (verified)
- `data/` contains: `archives/`, `cache/`, `episodic/`, `logs/`, `semantic/`, `working_state/`, `.gitkeep`
- `data/retrieval/` is absent
- `.gitignore` uses `data/*` glob with selective `!data/.gitkeep` exemptions

### Touch points
- `data/retrieval/` — create directory
- `data/retrieval/.gitkeep` — create placeholder file
- `.gitignore` — verify (and add if missing) that `data/retrieval/**` runtime artifacts are covered

### Implementation steps

**Step 1 — Create the directory and placeholder**
```
New-Item -ItemType Directory -Path data\retrieval -Force
New-Item -ItemType File -Path data\retrieval\.gitkeep -Force
```
Or equivalent shell. The directory must exist in the repository tree.

**Step 2 — Verify .gitignore coverage**
Open `.gitignore` and confirm the existing `data/*` glob covers `data/retrieval/` runtime
artifacts. The current `.gitignore` already contains:
```
data/*
!data/models.yaml.example
!data/.gitkeep
data/**/*.db
data/**/*.db-journal
...
```
The `data/*` glob suppresses the directory contents but does NOT suppress the `.gitkeep` file
unless a matching `!data/**/.gitkeep` exemption exists.

Add the following line to `.gitignore` under the `# Runtime data artifacts` section if not
already present:
```
!data/**/.gitkeep
```
This ensures all `.gitkeep` placeholders across `data/` subdirectories are committed without
committing runtime artifacts.

**Step 3 — Confirm no code changes required**
`backend/retrieval/hybrid_retriever.py` writes retrieval artifacts to paths configured at
runtime (not hardcoded to `data/retrieval/`). No source changes needed for this task.

### Acceptance criteria
- `data/retrieval/` directory exists in repository.
- `data/retrieval/.gitkeep` is committed.
- `.gitignore` exempts `.gitkeep` files under `data/` subdirectories.
- No other files modified.

### Evidence required
```
# Directory listing confirming presence
dir data\retrieval

# Confirm .gitignore has the exemption
Select-String -Path .gitignore -Pattern "gitkeep"
```
Expected: `data/retrieval/.gitkeep` visible; `.gitignore` shows at least one `!data/**/.gitkeep`
or equivalent exemption line.

## Task T12.3 — Frontend Decomposition: Extract `state/`, `styles/`, `utils/`

### Why
`App.jsx` is currently ~400 lines with state logic, inline style constants, and utility
functions co-located. `Project.md` §13 defines the following as distinct modules:
- `frontend/src/state/` — client-side state management
- `frontend/src/styles/` — theme, layout, global styles
- `frontend/src/utils/` — shared UI utility helpers

This task extracts existing logic into those modules with zero behavioral change. It reduces
future maintenance risk and aligns the repo layout with the architecture spec.

### Current state (verified)

`App.jsx` contains the following extractable units (confirmed by code review):

**Utility functions** (pure, no React dependencies):
- `truncatePreviewText(value, maxLength)` — text truncation helper
- `normalizePreviewItems(items)` — array slice/normalization helper
- `renderToolPreview(toolPreview)` — JSX renderer for tool result preview block
- `renderAssistantContent(content)` — JSX renderer wrapping ReactMarkdown

**Style constants** (inline style objects, reused across multiple components):
- Color palette constants: `#050810`, `#0a0e1a`, `#1a2332`, `#00d4ff`, `#e5f6ff`, `#ef4444`,
  `#b8deeb`, `#8fb6c2` — used in message bubbles, input bar, header, tool preview
- Common style shapes: message bubble base, code block, input field, send button

**State and handlers** (React state + event logic):
- `messages`, `taskId`, `finalState`, `isLoading`, `input`, `selectedFile` state
- `handleSend()` — full send/stream/upload dispatch logic
- `handleNewChat()` — chat reset logic
- Health polling effects (`getHealth`, `getDetailedHealth`)
- Derived display values: `modelIndicator`, `cacheIndicator`, `shortTaskId`

`SettingsPanel.jsx` and `WorkflowVisualizer.jsx` are **not touched** by this task — they are
already separate components and import from `../api/taskClient` directly.

### Touch points
- `frontend/src/App.jsx` — reduce to thin composition shell (target: ≤150 lines)
- new `frontend/src/utils/renderHelpers.jsx` — extract render utility functions
- new `frontend/src/styles/theme.js` — extract color/spacing constants
- new `frontend/src/state/useChatState.js` — extract chat state as a custom hook
- `frontend/package.json` — no changes expected; no new dependencies required

### Implementation steps

#### Step 4 — Create `frontend/src/state/useChatState.js`

Extract the chat state and event handlers from `App.jsx` into a custom React hook. This is a
`.js` file (returns a plain object, not JSX).

**Target file: `frontend/src/state/useChatState.js`**

The hook must encapsulate:
- State: `messages`, `taskId`, `finalState`, `isLoading`, `input`, `selectedFile`
- Refs: `inputRef`, `fileInputRef`, `messagesEndRef`
- Handlers: `handleSend`, `handleNewChat`
- Effects: scroll-to-bottom on messages change

The hook must **not** encapsulate:
- Health polling effects (these remain in `App.jsx` — they are system-level, not chat-level)
- `isSettingsOpen` / `setIsSettingsOpen` (panel toggle state, remains in App)

The hook must accept no required arguments and return all state/refs/handlers needed by
`App.jsx` to render the chat pane and input bar.

Imports required inside the hook:
```js
import { useEffect, useRef, useState } from 'react'
import { createOrContinueTaskStream, createOrContinueTaskUpload } from '../api/taskClient'
```

Return shape:
```js
return {
  messages,
  isLoading,
  input,
  setInput,
  selectedFile,
  setSelectedFile,
  taskId,
  finalState,
  inputRef,
  fileInputRef,
  messagesEndRef,
  handleSend,
  handleNewChat,
}
```

**Constraint:** No logic changes. The body of `handleSend` and `handleNewChat` must be
identical to the current `App.jsx` implementations — move, do not rewrite.
