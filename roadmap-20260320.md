# roadmap-20260320.md

## Milestone M19: Agentic Planning Depth + Multi-Step Task Quality
**Date:** 2026-03-20
**Status:** Draft — Pending Approval

---

### Goal

M19 improves the quality and depth of agentic task execution across four areas:

1. **Router intent coverage is thin** — `RouterNode` classifies into `chat`, `code`, `research` using simple keyword markers. `planning` and `writing` are common daily-use intents (task breakdown, document drafting, structured output) that receive no specialized routing and fall through to `chat` regardless of input.

2. **Plan decomposition is heuristic-only** — `build_constrained_plan()` splits on delimiters (newlines, semicolons, periods, "then", "next") and character count. This produces subtasks mechanically from punctuation rather than from semantic intent. Multi-sentence inputs produce poor or spurious subtask splits; genuine multi-step requests may not trigger planning at all.

3. **Per-subtask validation is absent** — `ValidatorNode` runs once after all subtasks are aggregated. A bad subtask output that produces an empty or error string is silently skipped during the loop and only fails at the terminal gate. There is no opportunity to log, skip, or partially recover a failing subtask.

4. **Subtask execution context is opaque** — `context["planning_subtasks"]` and `context["planning_aggregated_parts"]` are tracked but there is no per-subtask status record in the episodic trace. Debugging a multi-step failure requires reading raw DAG events to reconstruct what happened per subtask.

**What is already implemented (do not re-implement):**
- `RouterNode` — three-intent classification (`chat`, `code`, `research`), verified
- `build_constrained_plan()` — heuristic delimiter splitting with `MAX_SUBTASKS=3`, `PLANNING_MIN_INPUT_CHARS=80`, verified
- `compile_plan_to_workflow_graph()` — fixed graph compiler (currently ignores intent argument)
- EXECUTE phase subtask loop — iterates `execution_inputs`, resets context per subtask, aggregates outputs with `[Part N]` wrapping, deduplicates attachment-driven outputs, verified M11
- `ValidatorNode` — terminal output quality checks (empty, too short, model error markers), verified M13
- `context["planning_mode"]`, `context["planning_subtasks"]`, `context["planning_aggregated_parts"]` — already set in controller

**Design constraints:**
- Deterministic control — all routing and planning decisions are rule-based. No LLM call is introduced into the PLAN phase. The LLM remains confined to the EXECUTE phase nodes (`llm_worker`, `search_web`, `tool_call`).
- Minimal diff principle — changes are additive where possible. The existing subtask loop, graph structure, and DAG executor are preserved.
- `ValidatorNode` behavior is unchanged for the terminal validation pass. Per-subtask validation is an additional lightweight check inside the loop, not a replacement.
- `compile_plan_to_workflow_graph()` currently ignores its argument. M19 activates intent-aware graph selection using the existing argument slot — no API or signature change needed.

**Verified starting state (from `SYSTEM_INVENTORY.md` M18 entry and `CHANGE_LOG.md` 2026-03-19 10:55):**
- `backend/workflow/nodes/router_node.py` — keyword markers for `code` and `research`; all other input → `chat`; no `planning` or `writing` paths
- `backend/workflow/plan_compiler.py` — `build_constrained_plan()` splits on delimiters; `compile_plan_to_workflow_graph()` always returns fixed 4-node graph regardless of argument
- `backend/workflow/nodes/validator_node.py` — checks `llm_output` for empty, too-short, model-error-markers, explicit `llm_error`; single terminal pass only
- `backend/controller/controller_service.py` — EXECUTE loop iterates subtasks; no per-subtask status logged to episodic trace; `ValidatorNode` called once after aggregation
- Test baseline: use the actual passing count from the regression run at implementation time (latest cited evidence at roadmap authoring: `PASS_WITH_SKIPS` after M18)

## T19.1 — Workflow: Add `planning` and `writing` Intents to RouterNode

### Why

`RouterNode` returns `chat` for any input that is not `code` or `research`. Two common daily-use patterns fall into this residual bucket:

- **`planning`** — inputs that ask for a breakdown, plan, steps, tasks, schedule, outline, or agenda. These benefit from multi-step execution where each step is a focused subtask.
- **`writing`** — inputs that ask for a draft, write, compose, letter, email, report, essay, or document. These benefit from structured single-pass execution with a focused writing prompt rather than a generic chat response.

Both are detected by keyword markers, consistent with the existing pattern. Precedence order matters: `code` is checked first (unchanged), then `research`, then `planning`, then `writing`, then `chat` as the final fallback.

### Touch points

- `backend/workflow/nodes/router_node.py`:
  - Add marker tuples:
    ```python
    _PLANNING_MARKERS = (
        "plan", "planning", "break down", "breakdown", "steps", "step by step",
        "tasks", "task list", "schedule", "outline", "agenda",
    )
    _WRITING_MARKERS = (
        "write", "writing", "draft", "compose", "letter", "email",
        "report", "essay", "document", "summarize", "summary",
    )
    ```
  - Update `execute()` classification logic to check `planning` and `writing` after `research` and before `chat`:
    ```python
    if "code" in user_input:
        intent = "code"
    elif any(marker in user_input for marker in _RESEARCH_MARKERS):
        intent = "research"
    elif any(marker in user_input for marker in _PLANNING_MARKERS):
        intent = "planning"
    elif any(marker in user_input for marker in _WRITING_MARKERS):
        intent = "writing"
    else:
        intent = "chat"
    ```
  - No other changes to `RouterNode`

### Acceptance

- Input containing `"plan"` → `intent == "planning"`.
- Input containing `"write"` → `intent == "writing"`.
- Input containing `"search"` → `intent == "research"` (unchanged).
- Input containing `"code"` → `intent == "code"` (unchanged).
- Plain conversational input → `intent == "chat"` (unchanged).
- `code` check still takes precedence over all others.

- `backend/.venv/Scripts/python -m pytest tests/unit/test_nodes.py -q` — PASS
- `backend/.venv/Scripts/python scripts/validate_backend.py --scope unit` — PASS (no regressions)

## T19.2 — Workflow: Intent-Aware Graph Selection in `compile_plan_to_workflow_graph()`

### Why

`compile_plan_to_workflow_graph()` currently ignores its `plan_artifact` argument and always returns the same fixed 4-node graph. The argument was intentionally designed to accept intent/plan data — the docstring says "Converts plan steps into executable nodes" — but the implementation never materialized. M19 activates this by routing each intent to an appropriate graph:

- **`planning`** — same graph as `chat` (context_builder → llm_worker), but the planning intent triggers `build_constrained_plan()` in the controller to decompose into subtasks. The graph itself is unchanged; the planning behavior comes from the subtask loop.
- **`writing`** — same graph as `chat`. The writing intent signals a focused single-pass generation; the graph is identical to `chat` but downstream tooling can gate behavior on the intent value.
- **`research`** — existing `search_web` insertion handled by `_with_search_web_node()` in the controller; `compile_plan_to_workflow_graph()` does not need to change for this intent.
- **`code`** and **`chat`** — fixed 4-node graph (unchanged).

The practical change here is making `compile_plan_to_workflow_graph()` explicitly document and formally return the correct graph per intent rather than silently ignoring the argument. This also ensures the `planning` and `writing` intents do not accidentally use a graph intended for a different path if this function is extended in a future milestone.

### Touch points

- `backend/workflow/plan_compiler.py`:
  - Update `compile_plan_to_workflow_graph(plan_artifact: str | None) -> WorkflowGraph`:
    - Parse the `plan_artifact` argument as an intent string (strip, lowercase)
    - Define the base chat/planning/writing/code graph (4-node fixed: `router → context_builder → llm_worker → validator`) — same as current
    - For `"research"`: return the same base graph; the controller's `_with_search_web_node()` handles injection
    - For all intents (`chat`, `code`, `planning`, `writing`, `research`, and unknown/None): return the base graph
    - The function now explicitly dispatches on intent rather than ignoring the argument — this is a no-op in terms of graph shape today, but makes the routing explicit and testable
  - Add `SUPPORTED_INTENTS: frozenset[str] = frozenset({"chat", "code", "research", "planning", "writing"})` as a module-level constant, for use in tests and the controller

### Acceptance

- `compile_plan_to_workflow_graph("planning")` returns the same graph as `compile_plan_to_workflow_graph("chat")`.
- `compile_plan_to_workflow_graph("writing")` returns the same graph as `compile_plan_to_workflow_graph("chat")`.
- `compile_plan_to_workflow_graph(None)` returns the base graph without error.
- `compile_plan_to_workflow_graph("unknown_intent")` returns the base graph without error.
- `SUPPORTED_INTENTS` is importable and contains all five intent strings.
- All existing graph shapes are preserved — this is an explicit no-op for current graph structure.

- `backend/.venv/Scripts/python scripts/validate_backend.py --scope unit` — PASS (no regressions)

## T19.3 — Workflow: Improve `build_constrained_plan()` Trigger Heuristic

### Why

The current `build_constrained_plan()` trigger requires two conditions simultaneously: `len(input) >= 80 chars` AND `len(decomposed_segments) >= 2`. The decomposition splits on `\n`, `;`, `.`, ` then `, ` next `. This means:

- Short multi-step requests (under 80 chars) never trigger planning even if they are clearly multi-step.
- Sentences with periods produce spurious subtask splits ("Write a report on AI. Include examples." → three fragments including an empty one).
- `planning` intent inputs identified by T19.1 should more reliably trigger `planned` mode regardless of length.

M19 improves the heuristic in two ways:
1. **Intent-aware trigger**: if `context["intent"]` is `"planning"`, force `planned` mode regardless of character count threshold.
2. **Cleaner decomposition**: remove period (`.`) as a split delimiter — it produces too many false positives on normal prose. Retain `\n`, `;`, ` then `, ` next `.

`build_constrained_plan()` currently does not receive `intent` as an argument. The controller calls it after setting `context["intent"]`. Adding an optional `intent` parameter to `build_constrained_plan()` is the clean approach.

### Touch points

- `backend/workflow/plan_compiler.py`:
  - Update `build_constrained_plan(user_input: str, intent: str = "") -> dict[str, Any]`:
    - Add `intent: str = ""` parameter
    - Remove `.` from the list of split delimiters in `_decompose_user_input()`; retain `\n`, `;`, ` then `, ` next `
    - Update planning trigger: `planning_triggered` is `True` if either:
      - `str(intent).strip().lower() == "planning"` (intent-forced), OR
      - `len(normalized_input) >= PLANNING_MIN_INPUT_CHARS and len(decomposed) >= 2` (existing heuristic)
    - When `planning_triggered` due to intent but decomposition yields only one segment, still enter `planned` mode with a single subtask equal to the full input — this ensures `planning` intent always produces the dedicated single-pass structured execution path even for short inputs
  - No changes to `MAX_SUBTASKS` or `PLANNING_MIN_INPUT_CHARS` constants

- `backend/controller/controller_service.py`:
  - Update the `build_constrained_plan()` call to pass `intent`:
    ```python
    constrained_plan = build_constrained_plan(user_input, intent=str(context.get("intent", "")))
    ```

### Acceptance

- `build_constrained_plan("Plan my week", intent="planning")` → `mode == "planned"` (short input, intent-forced).
- `build_constrained_plan("Write a report on AI. Include examples.", intent="chat")` → `mode == "linear"` (period no longer splits; under threshold as single segment).
- `build_constrained_plan("Do X then do Y then do Z")` → `mode == "planned"` if length >= 80 chars, `linear` otherwise (existing heuristic preserved for non-planning intents).
- `build_constrained_plan("short input")` → `mode == "linear"` (unchanged fallback).
- `MAX_SUBTASKS` cap is still respected.

- `backend/.venv/Scripts/python scripts/validate_backend.py --scope unit` — PASS (no regressions)

## T19.4 — Controller: Per-Subtask Status Logging to Episodic Trace

### Why

The EXECUTE loop currently collects subtask outputs in `aggregated_subtask_outputs` but records nothing to the episodic trace about individual subtask outcomes. When a planned task fails or produces unexpected output, there is no trace of which subtask index produced what, making debugging opaque.

Adding a `dag_subtask_event` decision record per subtask — recording the subtask index, input text (truncated), output text (truncated), and success/skip status — provides the traceability required by `Project.md` §1 invariant 4 ("Every action produces replayable artifacts") without changing the subtask execution logic itself.

This is a pure logging addition inside the `planned` mode branch of the EXECUTE loop.

### Touch points

- `backend/controller/controller_service.py`:
  - Inside the EXECUTE loop, after the per-subtask node execution and before appending to `aggregated_subtask_outputs`, when `planning_mode == "planned"`, add:
    ```python
    subtask_output = str(context.get("llm_output", "")).strip()
    subtask_payload = {
        "subtask_index": index,
        "subtask_count": len(execution_inputs),
        "subtask_input_preview": str(execution_input)[:120],
        "subtask_output_preview": subtask_output[:120],
        "subtask_output_empty": not bool(subtask_output),
    }
    self.memory.log_decision(
        task_id=resolved_task_id,
        action_type="dag_subtask_event",
        content=json.dumps(subtask_payload, sort_keys=True, separators=(",", ":")),
        status="completed" if subtask_output else "empty_output",
    )
    ```
  - The record is written unconditionally for all subtasks (including those that produce empty output) so the trace is complete
  - `linear` mode tasks do not produce `dag_subtask_event` records — this event type is planning-only

### Acceptance

- After a `planned` mode task with 2 subtasks, `episodic.search_decisions(query="dag_subtask_event", task_id=...)` returns 2 records.
- Each record's `content` parses as JSON with `subtask_index`, `subtask_count`, `subtask_input_preview`, `subtask_output_preview`, `subtask_output_empty` fields.
- `subtask_output_empty == True` for a subtask that produced no output.
- `linear` mode tasks produce no `dag_subtask_event` records.
- Existing `dag_node_event` records are unaffected.

### Evidence

- `backend/.venv/Scripts/python -m pytest tests/unit/test_controller_service_integration.py -q` — PASS
- `backend/.venv/Scripts/python scripts/validate_backend.py --scope unit` — PASS (no regressions)

## T19.5 — Controller: Per-Subtask Output Validation Gate

### Why

Currently a subtask that produces empty output (model error, missing model, escalation failure) is silently skipped — `subtask_output` is not appended to `aggregated_subtask_outputs` only because of the `if subtask_output:` guard. This means partial failures in a multi-step task are invisible until terminal validation, which then fails on the aggregated empty result.

A lightweight per-subtask gate after each subtask execution catches this early, sets a dedicated context key (`planning_subtask_failures`), and allows the loop to continue collecting remaining valid outputs rather than continuing blindly. If any subtask fails the gate, the final aggregated output still proceeds to terminal `ValidatorNode` — the gate is informational and does not short-circuit the loop.

This follows the existing fail-closed pattern: a completely empty aggregated result still fails terminal validation. The per-subtask gate adds diagnostic visibility and populates `context["planning_subtask_failures"]` for trace and API consumers.

### Touch points

- `backend/controller/controller_service.py`:
  - Before the EXECUTE loop, initialize: `context["planning_subtask_failures"] = []`
  - Inside the loop, after collecting `subtask_output` (but before appending to `aggregated_subtask_outputs`), when `planning_mode == "planned"`, add a lightweight check:
    ```python
    if not subtask_output:
        context["planning_subtask_failures"].append({
            "subtask_index": index,
            "subtask_input_preview": str(execution_input)[:120],
            "reason": str(context.get("llm_error", "empty_output")),
        })
    ```
  - The existing `if subtask_output: aggregated_subtask_outputs.append(subtask_output)` guard is unchanged — empty subtask outputs are still not aggregated
  - For `linear` mode, `context["planning_subtask_failures"]` is initialized to `[]` and remains empty (not populated)
  - No change to the terminal `ValidatorNode` behavior or the `_fail()` path

### Acceptance

- After a planned task where one subtask produces empty output, `context["planning_subtask_failures"]` contains one entry with `subtask_index`, `subtask_input_preview`, `reason` fields.
- After a planned task where all subtasks succeed, `context["planning_subtask_failures"] == []`.
- `linear` mode tasks have `context["planning_subtask_failures"] == []` always.
- Terminal `ValidatorNode` still runs on the aggregated output regardless of `planning_subtask_failures` content.
- A fully failed planned task (all subtasks empty) still reaches terminal validation and fails via the existing `"validation_failed"` path.

### Evidence

- `backend/.venv/Scripts/python -m pytest tests/unit/test_controller_service_integration.py -q` — PASS
