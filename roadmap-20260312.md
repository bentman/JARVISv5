### Goal

M15 closes G-B holistically. The gap has five sub-problems that must be addressed together to avoid creating new tech-debt: no escalation policy layer, no external API provider registry, `.env`/`.env.example` out of sync on API keys, no UI provider selection surface, and no redaction on the escalation path. Implementing policy alone without the key registry or UI surface would leave the system partially configured with no operator-visible control — the same failure mode that created the G-A search controls debt.

`Project.md` §3.3, §6.3, and §9.2 require escalation policy with cloud model fallback, configurable per-provider budget limits, privacy redaction before external API calls, and environment-variable-driven configuration. §3.5 and §11.2 require configurable settings surfaced in the UI including provider controls.

**Design principle for this and all future milestones:** When a gap has env, backend, and UI dimensions, all three must be closed in the same milestone. Partial closure is deferred debt, not incremental progress.

## T15.1 — `.env` / `.env.example` Parity + API Key Registry

### Why

The four canonical cloud model provider API keys are absent from both `.env.example` and `settings.py`. `Project.md` §9.2 states all configuration is environment-variable-driven. `.env.example` is the canonical template — any key that will ever be read by the system must appear there with a blank default. `.env` must stay in sync with `.env.example`'s structure so operators can compare them without guesswork.

A dedicated `backend/config/api_keys.py` module keeps API key concerns separate from general operational settings in `settings.py`. This is the same separation principle applied between `backend/search/policy.py` and `settings.py`. API keys are credentials, not configuration — they warrant their own module with a clear read-only projection.

### Touch points

- `.env.example` — add under a new `# API Keys and External Services` section:
  ```
  ANTHROPIC_API_KEY=
  GEMINI_API_KEY=
  GROK_API_KEY=
  OPENAI_API_KEY=
  ```
- `.env` — add the same four keys with blank values (preserve any existing `TAVILY_API_KEY` entry in its existing search section; these four are cloud model providers, not search)
- `backend/config/api_keys.py` (new file):
  - `ApiKeyRegistry` — reads the four `*_API_KEY` vars from environment at instantiation time (not from `settings.py`); each key stored as `str`, default `""`
  - `get_configured_providers() -> list[str]` — returns sorted list of provider name strings whose key is non-empty (e.g. `["anthropic", "openai"]`); empty list if none configured
  - `get_api_key(provider: str) -> str` — returns key for named provider or `""` if absent/unconfigured; provider names: `"anthropic"`, `"gemini"`, `"grok"`, `"openai"`
  - `SUPPORTED_PROVIDERS: frozenset[str]` — canonical set of supported provider names; used for validation in T15.3
  - No writes; no side effects; deterministic for same environment state

### Acceptance

- `.env.example` and `.env` both contain all four `*_API_KEY` entries.
- `get_configured_providers()` returns only providers with non-empty keys.
- `get_api_key("openai")` returns `""` when `OPENAI_API_KEY` is unset.
- `get_configured_providers()` is deterministic for the same environment state.
- No API key value is ever logged, serialized to `GET /settings`, or included in any response body.

## T15.2 — Escalation Policy Module

### Why

The policy logic for deciding whether to escalate belongs in a dedicated module, not inline in the controller — the same separation pattern as `backend/search/policy.py`. The policy is a pure function: explicit inputs in, typed decision artifact out, no side effects.

The module must consult `get_configured_providers()` from T15.1 as a gate — a provider with no configured API key cannot be approved regardless of other settings. This prevents policy from approving escalation to a provider the system cannot actually reach.

### Touch points

- `backend/models/escalation_policy.py` (new file):
  - `EscalationTrigger` enum — `local_model_missing` (registry returned None), `local_model_path_error` (ensure_model_present raised), `resource_exhaustion` (hardware state signals constraint)
  - `EscalationPolicyRequest` Pydantic model — fields: `trigger: EscalationTrigger`, `allow_escalation: bool`, `provider: str`, `provider_key_configured: bool`, `budget_usd: float`, `estimated_cost_usd: float`
  - `decide_escalation(request: EscalationPolicyRequest) -> tuple[bool, dict]` — pure decision function; gates in order:
    1. `allow_escalation=False` → deny, `code="permission_denied"`
    2. `provider` empty → deny, `code="provider_not_configured"`
    3. `provider_key_configured=False` → deny, `code="provider_key_missing"`
    4. `budget_usd <= 0.0` → deny, `code="budget_not_allocated"`
    5. `estimated_cost_usd > budget_usd` → deny, `code="budget_exceeded"`
    6. All pass → allow, `path="escalated"`
  - Result dict always includes `code`, `reason`, `path` (`"local"` / `"escalated"` / `"blocked"`), `trigger`, `provider`, `provider_key_configured`
  - `EscalationProviderBase` abstract class — `name: str`; `execute(prompt: str, max_tokens: int, seed: int | None) -> tuple[bool, str, str]` (ok, output, error); deterministic control invariant: called only when `decide_escalation()` returns allow
  - `StubEscalationProvider(EscalationProviderBase)` — returns fixed deterministic output; used in tests only; `name = "stub"`
- `backend/models/__init__.py` — export all new symbols

### Acceptance

- `decide_escalation()` returns correct allow/deny for all six gate paths.
- `provider_key_missing` gate fires correctly when `provider_key_configured=False` even if other gates would pass.
- `StubEscalationProvider.execute()` is deterministic for identical inputs.
- No live network calls in any code in this module.

### Evidence

- `backend/.venv/Scripts/python -m pytest tests/unit/test_escalation_policy.py -q` — PASS
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS (no regressions)

## T15.3 — Escalation Settings Fields

### Why

Three new settings govern escalation runtime behavior: a master gate, a provider selector, and a budget ceiling. These belong in `settings.py` alongside the existing `ALLOW_EXTERNAL_SEARCH` and `ALLOW_PAID_SEARCH` controls. The `ESCALATION_PROVIDER` validator must cross-check against `SUPPORTED_PROVIDERS` from T15.1 — an unknown provider name is rejected at validation time, not silently ignored.

### Touch points

- `backend/config/settings.py`:
  - Add `ALLOW_MODEL_ESCALATION: bool = False`
  - Add `ESCALATION_PROVIDER: str = ""`
  - Add `ESCALATION_BUDGET_USD: float = 0.0`
  - Extend `SafeConfigProjection` TypedDict with all three fields plus `escalation_configured_providers: list[str]` (populated at projection time from `get_configured_providers()`)
  - Extend `get_safe_config_projection()` accordingly
  - Add `"allow_model_escalation"` and `"escalation_provider"` to `EDITABLE_SETTINGS_ENV_KEYS` (budget is read-only via API, consistent with `DAILY_BUDGET_USD`)
  - Add `normalize_escalation_provider()` validator — accepts `""` or any value in `SUPPORTED_PROVIDERS`; rejects all others
  - Extend `serialize_editable_setting_value()` and `settings_update_restart_semantics()` for the two new editable keys (`allow_model_escalation` hot-applied; `escalation_provider` hot-applied)
- `backend/api/schemas.py`:
  - Add to `SettingsResponse`: `allow_model_escalation: bool | None`, `escalation_provider: str | None`, `escalation_budget_usd: float | None`, `escalation_configured_providers: list[str] | None`
  - Add to `SettingsUpdateRequest`: `allow_model_escalation: bool | None`, `escalation_provider: str | None` with validators
- `.env.example` — add `ALLOW_MODEL_ESCALATION=false`, `ESCALATION_PROVIDER=`, `ESCALATION_BUDGET_USD=0.0`
- `.env` — add same three keys with same safe defaults

### Acceptance

- `GET /settings` includes `allow_model_escalation`, `escalation_provider`, `escalation_budget_usd`, and `escalation_configured_providers` (populated from `get_configured_providers()`).
- `POST /settings {"escalation_provider": "unknown"}` returns `422`.
- `POST /settings {"escalation_provider": "openai"}` succeeds only if `"openai"` is in `SUPPORTED_PROVIDERS`.
- `ESCALATION_BUDGET_USD` appears in GET response but POST rejects it (not in `SettingsUpdateRequest`).

## T15.4 — Controller Escalation Integration + Redaction

### Why

The controller currently handles both `select_model() is None` and `ensure_model_present()` `RuntimeError` with the same silent fallback. Neither path consults policy, attempts escalation, produces structured failure artifacts, or applies redaction. `Project.md` §6.3 explicitly requires *"Automatic redaction before external API calls."* The `RedactionEngine` exists in `backend/security/` but is not applied on this path.

### Touch points

- `backend/controller/controller_service.py`:
  - Import `EscalationTrigger`, `EscalationPolicyRequest`, `EscalationProviderBase`, `decide_escalation` from `backend.models.escalation_policy`
  - Import `ApiKeyRegistry` from `backend.config.api_keys`
  - Replace both `local_model_missing` fallback blocks with the policy-evaluation sequence:
    1. Build `EscalationPolicyRequest` from `Settings()` fields and `ApiKeyRegistry().get_api_key(provider)`
    2. Call `decide_escalation()`
    3. Deny path: set `context["escalation_status"] = "denied"`, `context["escalation_code"]`, `context["escalation_reason"]`; keep `skip_llm=True`; set human-readable `llm_output` message
    4. Allow path: apply `RedactionEngine` to prompt; look up provider in `_ESCALATION_PROVIDER_REGISTRY`; call `provider.execute()`; on success set `context["escalation_status"] = "escalated"`, clear `skip_llm`; on failure set `context["escalation_status"] = "failed"`, `context["escalation_error"]`
  - Set `context["escalation_status"] = "not_attempted"` when local model is found successfully
  - Add `_ESCALATION_PROVIDER_REGISTRY: dict[str, EscalationProviderBase] = {}` — empty for production in this milestone; `"stub"` entry added only in tests via monkeypatch
  - No changes to `plan_compiler.py`, `dag_executor.py`, or workflow nodes

### Acceptance

- `escalation_status == "not_attempted"` when local model is found.
- `escalation_status == "denied"` + correct `escalation_code` for each policy gate (disabled, no provider, key missing, zero budget).
- `escalation_status == "escalated"` + `llm_output` from stub when all gates pass and stub provider registered.
- `escalation_status == "failed"` when provider raises.
- Redaction applied to prompt before any allow-path provider call; context includes `escalation_redaction_applied: bool`.
- `ValidatorNode` still runs on escalated output.
- Existing chat / code / research paths unaffected.

### Evidence

- `backend/.venv/Scripts/python -m pytest tests/unit/test_controller_service_integration.py -q` — PASS
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS (no regressions)

## T15.6 — UI Provider Selection Dropdown

### Why

`Project.md` §3.5 and §11.2 require provider controls in the Settings panel. Without a UI surface, `ESCALATION_PROVIDER` can only be set by editing `.env` directly. `escalation_configured_providers` from `GET /settings` gives the frontend exactly the data it needs: the list of providers with valid keys. The dropdown should render only configured providers — selecting a provider with no key is not a valid operation.

### Touch points

- `frontend/src/components/SettingsPanel.jsx` (or equivalent settings component):
  - Fetch `escalation_configured_providers` from `GET /settings` on panel open (same fetch that already retrieves budget and other settings)
  - Render an escalation section with:
    - `ALLOW MODEL ESCALATION` toggle — maps to `allow_model_escalation`; `POST /settings` on change
    - `ESCALATION PROVIDER` dropdown — populated from `escalation_configured_providers`; disabled when toggle is off or list is empty; shows placeholder "No providers configured" when empty; `POST /settings {"escalation_provider": value}` on selection
  - Follow existing panel styling (cyan accent, panel background `#1a2332`, consistent with current Settings & Budget section appearance)
- `frontend/src/api/taskClient.js` (or equivalent API client) — no new function required if settings fetch already uses existing `getSettings()` / `updateSetting()` helpers; verify those helpers exist and cover the new fields

### Acceptance

- Dropdown renders only providers returned in `escalation_configured_providers`.
- Dropdown is disabled when `allow_model_escalation` is false or `escalation_configured_providers` is empty.
- Selecting a provider calls `POST /settings {"escalation_provider": "<name>"}` and re-fetches settings.
- Toggle calls `POST /settings {"allow_model_escalation": <bool>}`.
- No API key value is rendered or transmitted from the frontend.
- `npm --prefix frontend run build` — PASS (no build errors)
