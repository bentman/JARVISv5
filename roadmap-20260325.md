## T20.1 — Frontend API Client: Voice Functions in `taskClient.js`

### Why

`POST /voice/transcribe` accepts `{"audio_path": "<path>"}` and returns `{"transcript": "...", "model_id": "...", "profile": "...", "hardware": "..."}`. `POST /voice/speak` accepts `{"text": "..."}` and returns `{"audio_path": "...", "model_id": "...", "profile": "...", "hardware": "..."}`. Neither has a client binding in `taskClient.js`. Both are needed before `VoicePanel` can call them.

**Audio upload design decision (Agent must choose one and commit):**

The existing `POST /voice/transcribe` accepts `{"audio_path": "<server-side-path>"}` — a path already on the server filesystem. Browser-recorded audio is a `Blob` in memory and has no server-side path. Two options:

- **Option A — Multipart upload route (preferred):** Add `POST /voice/transcribe/upload` accepting `multipart/form-data` with an `audio` file field. Backend writes the upload to `data/voice/upload-<uuid>.<ext>`, calls `controller.transcribe(temp_path)`, cleans up, returns `VoiceTranscribeResponse`. Pattern is identical to `/task/upload`. `transcribeAudio(blob)` POSTs the blob as `FormData`. Requires one new backend route and one new test.
- **Option B — Base64/data-URI:** `transcribeAudio(blob)` encodes as base64 data URI and posts `{"audio_path": "<data-uri>"}`. `stt_provider.py` detects a data URI, writes to a temp file, and proceeds. Zero new routes. More invasive to the provider.

**Option A is the recommended path** — it follows the established `/task/upload` pattern, keeps provider code clean, and produces a testable route. The Agent must choose one option, implement it consistently across T20.1 and any backend touch, and document the choice in the CHANGE_LOG entry.

### Touch points

- `frontend/src/api/taskClient.js`:
  - Add `transcribeAudio(audioBlob)`:
    - Posts audio to the backend transcription endpoint (either `POST /voice/transcribe` with a `multipart/form-data` upload, or the base64/data-URI approach if zero backend change is chosen)
    - Returns `{ transcript, model_id, profile, hardware }`
    - Throws on non-OK response with status and body text
  - Add `speakText(text)`:
    - `POST /voice/speak` with `{"text": text}`
    - Returns `{ audio_path, model_id, profile, hardware }`
    - Throws on non-OK response with status and body text
  - Follow existing export and error-handling patterns in `taskClient.js`

- **If Option A (multipart upload route) is chosen:**
- `backend/api/main.py`: add `POST /voice/transcribe/upload` accepting `multipart/form-data` with `audio` file field; write upload to `data/voice/upload-<uuid>.<ext>`; call `controller.transcribe(temp_path)`; clean up temp file on success and failure; return `VoiceTranscribeResponse`
- `backend/api/schemas.py`: no new schema needed — reuse `VoiceTranscribeResponse`
- `tests/unit/test_api_voice_transcribe_upload.py`: new test file covering success path and 503 path for the upload route (do not modify the existing `test_api_voice_transcribe.py`)
  - **If Option B (base64/data-URI) is chosen:**
    - `backend/voice/stt_provider.py`: detect `data:` prefix in `audio_path`; decode and write to temp file; proceed with temp path; clean up after transcription
    - `tests/unit/test_voice_stt_provider.py`: add test covering data-URI input path

### Acceptance

- `transcribeAudio(blob)` posts audio to the backend and returns a parsed transcript payload.
- `speakText("hello")` posts to `POST /voice/speak` and returns a parsed audio path payload.
- Both throw on non-OK responses.
- `npm --prefix frontend run build` — PASS

## T20.2 — Frontend Utility: Audio Recorder

### Why

`VoicePanel` needs to capture mic audio from the browser. A self-contained `audioRecorder.js` utility isolates `MediaRecorder` usage from the panel component, keeping `VoicePanel.jsx` focused on UI state. The recorder produces a `Blob` that `transcribeAudio()` consumes.

### Touch points

- `frontend/src/utils/audioRecorder.js`:
  - Export `createAudioRecorder()` returning an object with:
    - `start()` — requests mic permission, starts `MediaRecorder`, collects chunks
    - `stop()` — stops `MediaRecorder`, returns a `Promise<Blob>` resolving to the recorded audio blob
  - Use `MediaRecorder` with `audio/webm` or `audio/wav` as available in the browser
  - If mic permission is denied or `MediaRecorder` is unavailable, `start()` throws with a descriptive error string
  - No external dependencies; browser APIs only

### Acceptance

- `createAudioRecorder()` is importable from `frontend/src/utils/audioRecorder.js`.
- `start()` and `stop()` are present on the returned object.
- `npm --prefix frontend run build` — PASS

### Evidence

- `npm --prefix frontend run build` — PASS

## T20.3 — Frontend Component: `VoicePanel.jsx`

### Why

No voice UI exists. `VoicePanel.jsx` provides: record/stop mic capture, transcript display, speak text input, and speak action. It follows the existing docked-panel pattern established by `MemoryPanel.jsx` and `SettingsPanel.jsx` — same styling, same `isOpen`/`isDocked`/`onClose` prop contract, same close button placement.

### Touch points

- `frontend/src/components/VoicePanel.jsx`:

  **Props:** `{ isOpen, isDocked, onClose, onTranscriptResult }`
  - `isOpen` — if false, return null (same as MemoryPanel)
  - `isDocked` — layout flag (same as MemoryPanel/SettingsPanel)
  - `onClose` — close button callback
  - `onTranscriptResult({ transcript })` — called when transcription succeeds; allows App to inject transcript into the chat input (same pattern as `onReferenceResult` in MemoryPanel)

  **STT section — Record:**
  - `Record` button: calls `createAudioRecorder().start()`; button label changes to `Stop` while recording
  - On stop: calls `transcribeAudio(blob)` from T20.1
  - On success: displays transcript text in a read-only result area; calls `onTranscriptResult({ transcript })`
  - On error: displays inline error message
  - Loading state during transcription

  **TTS section — Speak:**
  - Text input for speak text
  - `Speak` button: calls `speakText(text)` from T20.1
  - On success: `VoiceSpeakResponse` returns `audio_path` — a server-side filesystem path (e.g. `data/voice/tts-<uuid>.wav`). The backend has no static file serving route. For M20, render the `audio_path` as a read-only text display confirming synthesis succeeded. **Do not attempt `<audio src=...>` playback** — no audio serving endpoint exists and the path is not browser-accessible. Audio playback via browser is deferred to a future slice (requires a `GET /voice/audio/{filename}` serving route or static file mount, which is out of scope for M20).
  - On error: displays inline error message
  - Loading state during synthesis

  **Styling:** Follow existing panel CSS-in-JS patterns from `MemoryPanel.jsx` — dark background, cyan accent borders, same font sizes and button styles.

  **No changes** to `MemoryPanel.jsx`, `SettingsPanel.jsx`, or `WorkflowVisualizer.jsx`.

### Acceptance

- `VoicePanel` renders with record/stop control, transcript display, speak input, and speak button.
- Errors display inline per section without affecting the other section.
- `onTranscriptResult` is called on successful transcription.
- `npm --prefix frontend run build` — PASS

### Evidence

- `npm --prefix frontend run build` — PASS

## T20.4 — Frontend App: Wire `VoicePanel` into `App.jsx`

### Why

`App.jsx` manages all panels via `activePanel` state with a one-panel-at-a-time pattern. `VoicePanel` must be added to this pattern. A `Voice` button goes in the header alongside `Memory` and `Workflow` (second row, right side). Clicking it toggles `activePanel === 'voice'` and closes any other panel.

### Touch points

- `frontend/src/App.jsx`:
  - Import `VoicePanel` from `./components/VoicePanel`
  - Add `openVoicePanel` handler following the exact pattern of `openMemoryPanel` and `openWorkflowPanel`
  - Add `Voice` button in the header second row alongside `Memory` and `Workflow` — same `appStyles.headerButton` style
  - Add `activePanel === 'voice'` branch in the docked panel region:
    ```jsx
    {activePanel === 'voice' ? (
      <VoicePanel
        isOpen={true}
        isDocked={true}
        onClose={closePanels}
        onTranscriptResult={({ transcript }) => {
          setInput((previous) => {
            const current = String(previous || '')
            if (!current.trim()) return transcript
            return `${current}\n${transcript}`
          })
          inputRef.current?.focus()
        }}
      />
    ) : null}
    ```
  - No other changes to `App.jsx`

### Acceptance

- `Voice` button appears in the header.
- Clicking it opens `VoicePanel` in the docked region; other panels close.
- Clicking again closes it.
- Transcript result from `onTranscriptResult` is injected into the chat input.
- `npm --prefix frontend run build` — PASS
