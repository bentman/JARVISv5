# roadmap-20260319.md

## Milestone M18: Memory Recall + Semantic Retrieval UX
**Date:** 2026-03-19
**Status:** Draft — Pending Approval

---

### Goal

M18 makes the semantic memory layer useful to the user and observable in the system. The infrastructure is complete and producing data (bugfix-04 wired semantic writes on the validated path), but four gaps prevent effective daily use:

1. **No delete surface** — `SemanticMemory` has no delete method; stale or incorrect entries accumulate permanently.
2. **Retrieval is a black box** — `ContextBuilderNode` injects retrieved context into the prompt but sets no telemetry key; the user and the workflow visualizer have no visibility into what was injected or why.
3. **Memory panel is minimal** — results show snippets and a Reference button but no relevance scores, no source-type filter, no delete action, and no indication of whether a result influenced a recent task.
4. **Retrieval parameters are hardcoded** — `RetrievalConfig` defaults (`max_results=10`, `min_final_score_threshold=0.0`, `time_decay_tau_hours=24.0`) cannot be adjusted without code changes.

**What is already implemented (do not re-implement):**
- `SemanticMemory.add_text()`, `search()`, `search_text()` — FAISS + SQLite, verified M7
- `HybridRetriever.retrieve()` — three-source ranked retrieval (working state, semantic, episodic), verified M7
- `ContextBuilderNode._inject_retrieved_context()` — injects top results as system message into prompt, verified M12
- `GET /memory/search` — returns `semantic_results` + `episodic_results` with scores and metadata, verified M13
- `MemoryPanel.jsx` — search input, result cards with Reference button, verified M13

**Design constraints:**
- The semantic delete operation removes the SQLite metadata row and rebuilds the FAISS index from remaining rows. This is the only safe approach given FAISS `IndexFlatL2` does not support in-place removal. The rebuild is triggered in-process; for the expected data volumes of a personal assistant, this is acceptable.
- `ContextBuilderNode` already injects retrieved context via `_inject_retrieved_context()`. The telemetry key added in T18.2 must not change the injection behavior — it is additive context only.
- Retrieval configuration fields added to `Settings` in T18.3 are applied at `ContextBuilderNode` construction time in the controller, not as runtime hot-swappable knobs. They are hot-applied settings (no restart required) but take effect on the next task run.
- No changes to `HybridRetriever` scoring logic or `RetrievalConfig` defaults — the new settings fields override defaults when present.

**Verified starting state (from `SYSTEM_INVENTORY.md` M17 entry and `CHANGE_LOG.md` 2026-03-19 06:52):**
- `backend/memory/semantic_store.py` — `SemanticMemory`: `add_text()`, `search()`, `search_text()` present; no `delete()` method
- `backend/memory/memory_manager.py` — `store_knowledge()`, `retrieve_knowledge()`, `get_relevant_context()` present; no delete method
- `backend/api/main.py` — `GET /memory/search` present; no `DELETE /memory/semantic/{id}` route
- `backend/workflow/nodes/context_builder_node.py` — `_inject_retrieved_context()` injects results silently; sets no `retrieved_context_injected` or equivalent context key
- `backend/retrieval/retrieval_types.py` — `RetrievalConfig` with hardcoded defaults; not read from `Settings`
- `frontend/src/components/MemoryPanel.jsx` — search + result cards with Reference button; no score display, no filter, no delete
- `frontend/src/api/taskClient.js` — `searchMemory()` present; no `deleteMemoryEntry()` function
- Test baseline: use the actual passing count from the regression run at implementation time (latest cited evidence at roadmap authoring: `PASS_WITH_SKIPS` after M17)

- Embedding model selection or swap

## T18.1 — Backend: Semantic Memory Delete

### Why

`SemanticMemory` uses `IndexFlatL2` which does not support in-place vector removal. The only correct approach is to delete the SQLite metadata row and rebuild the FAISS index from the remaining rows via the existing `_rebuild_index_from_db()` method, then persist. The row's `id` (SQLite primary key) is already returned by `search_text()` as part of metadata and surfaced in `GET /memory/search` responses — it is the natural delete key.

`MemoryManager` wraps `SemanticMemory` and must expose `delete_knowledge(entry_id: int)`. The API route `DELETE /memory/semantic/{id}` provides the user-facing surface.

### Touch points

- `backend/memory/semantic_store.py`:
  - Add `delete(entry_id: int) -> bool`:
    - Delete the row with `id = entry_id` from the `embeddings` table
    - Call `_rebuild_index_from_db()` to reconstruct the FAISS index without the removed vector
    - Call `_persist_index_best_effort()` to write the updated index to disk
    - Return `True` if a row was deleted, `False` if no row matched (not an error)
  - Note: `vector_id` values in the FAISS index are positional (0-based insertion order). After rebuild, vector IDs for remaining rows are reassigned sequentially. The rebuild also updates the FAISS index in-memory; no stale references remain.

- `backend/memory/memory_manager.py`:
  - Add `delete_knowledge(entry_id: int) -> bool`:
    - Delegates to `self.semantic.delete(entry_id)`
    - Returns the bool result

- `backend/api/schemas.py`:
  - Add `MemoryDeleteResponse` model:
    ```python
    class MemoryDeleteResponse(BaseModel):
        deleted: bool
        entry_id: int
    ```

- `backend/api/main.py`:
  - Add route:
    ```python
    @app.delete("/memory/semantic/{entry_id}", response_model=MemoryDeleteResponse)
    def delete_semantic_memory(entry_id: int) -> MemoryDeleteResponse:
    ```
  - Raises `404` if `deleted=False` (entry not found)
  - Uses `_build_memory_manager(Settings())` consistent with other memory routes

### Acceptance

- `DELETE /memory/semantic/{id}` returns 200 with `{"deleted": true, "entry_id": id}` for a valid existing entry.
- `DELETE /memory/semantic/{id}` returns 404 for a non-existent id.
- Subsequent `GET /memory/search` for the deleted entry's content does not return it.
- FAISS index is rebuilt and persisted after deletion.
- `MemoryManager.delete_knowledge(id)` returns `True` when row exists, `False` when not.

## T18.2 — Backend: Retrieval Telemetry in Context

### Why

`ContextBuilderNode._inject_retrieved_context()` injects a "Retrieved Context" system message when results are found, but sets no context key to record what was injected. The workflow telemetry (`GET /workflow/{task_id}`) and any downstream diagnostic surface have no way to know whether retrieval fired, how many results were injected, or what their scores were. This makes the system opaque when debugging poor LLM responses that should have used stored knowledge.

The change is additive: a single context key is set after injection. No change to injection logic, scoring, or prompt structure.

### Touch points

- `backend/workflow/nodes/context_builder_node.py`:
  - In `_inject_retrieved_context()`, after the retrieval results are obtained and before returning:
    - Always set `context["retrieved_context_injected"]` — a list of dicts, one per injected result, each containing:
      ```python
      {
          "source": result.source.value,         # "semantic" | "episodic" | "working_state"
          "final_score": round(result.final_score, 4),
          "content_length": len(result.content),
          "vector_id": result.metadata.get("vector_id"),   # None for non-semantic
      }
      ```
    - If `self.retriever is None` or no results were found, set `context["retrieved_context_injected"] = []`
  - This key is set unconditionally so downstream code can always check it without `get()` guards

### Acceptance

- After a task run where retrieval fires, `context["retrieved_context_injected"]` is a non-empty list of dicts with the fields above.
- After a task run where retrieval returns no results, `context["retrieved_context_injected"] == []`.
- When `retriever is None`, `context["retrieved_context_injected"] == []`.
- Existing prompt injection behavior is unchanged — the retrieval message content is identical to before.
- `GET /workflow/{task_id}` node events already record context via `dag_node_event`; the new key flows through automatically.

## T18.3 — Backend: Configurable Retrieval Parameters

### Why

`RetrievalConfig` fields that affect user-visible behavior — how many results are retrieved (`max_results`), the minimum score gate (`min_final_score_threshold`), and the recency decay half-life (`time_decay_tau_hours`) — are hardcoded defaults. Users with different usage patterns (dense history vs sparse, short-lived vs long-lived knowledge) benefit from being able to tune these. Exposing them as `Settings` fields and wiring them into the `RetrievalConfig` constructed in the controller closes this gap with minimal invasiveness.

All three are hot-applied settings (no restart required). They take effect on the next task execution after being saved.

### Touch points

- `backend/config/settings.py`:
  - Add to `Settings`:
    ```python
    RETRIEVAL_MAX_RESULTS: int = 10
    RETRIEVAL_MIN_SCORE: float = 0.0
    RETRIEVAL_TIME_DECAY_TAU_HOURS: float = 24.0
    ```
  - Add to `SafeConfigProjection` TypedDict: `retrieval_max_results: int`, `retrieval_min_score: float`, `retrieval_time_decay_tau_hours: float`
  - Add to `get_safe_config_projection()` accordingly
  - Add to `EDITABLE_SETTINGS_ENV_KEYS`:
    ```python
    "retrieval_max_results": "RETRIEVAL_MAX_RESULTS",
    "retrieval_min_score": "RETRIEVAL_MIN_SCORE",
    "retrieval_time_decay_tau_hours": "RETRIEVAL_TIME_DECAY_TAU_HOURS",
    ```
  - Add serialization for all three in `serialize_editable_setting_value()`:
    - `retrieval_max_results`: `str(max(1, int(value)))`
    - `retrieval_min_score`: validate `0.0 <= float(value) <= 1.0`, serialize as string
    - `retrieval_time_decay_tau_hours`: validate `> 0.0`, serialize as string
  - Both new float fields are hot-applied; `retrieval_max_results` is also hot-applied

- `backend/api/schemas.py`:
  - Add to `SettingsResponse`: `retrieval_max_results: int | None = None`, `retrieval_min_score: float | None = None`, `retrieval_time_decay_tau_hours: float | None = None`
  - Add to `SettingsUpdateRequest`: same three fields with appropriate types and `None` defaults
  - Add field validators for range constraints matching the serialization rules above

- `backend/api/main.py`:
  - Thread the three new fields through both `GET /settings` and `POST /settings` handlers (same pattern as all other settings fields)

- `backend/controller/controller_service.py`:
  - When constructing `ContextBuilderNode`, pass a `RetrievalConfig` built from current `Settings`:
    ```python
    retrieval_config = RetrievalConfig(
        max_results=int(settings.RETRIEVAL_MAX_RESULTS),
        min_final_score_threshold=float(settings.RETRIEVAL_MIN_SCORE),
        time_decay_tau_hours=float(settings.RETRIEVAL_TIME_DECAY_TAU_HOURS),
    )
    context_builder_node = ContextBuilderNode(retrieval_config=retrieval_config, ...)
    ```
  - All other `RetrievalConfig` fields retain their defaults

- `.env.example` — add under a `# Retrieval` comment:
  ```
  RETRIEVAL_MAX_RESULTS=10
  RETRIEVAL_MIN_SCORE=0.0
  RETRIEVAL_TIME_DECAY_TAU_HOURS=24.0
  ```
- `.env` — add same three keys with same defaults

### Acceptance

- `GET /settings` returns `retrieval_max_results`, `retrieval_min_score`, `retrieval_time_decay_tau_hours`.
- `POST /settings {"retrieval_max_results": 5}` persists and returns updated value.
- `POST /settings {"retrieval_min_score": 1.1}` returns 422 (out of range).
- `POST /settings {"retrieval_time_decay_tau_hours": 0}` returns 422 (must be > 0).
- A task run after updating `retrieval_max_results=2` injects at most 2 retrieval results.
- `.env.example` and `.env` contain all three new keys.

## T18.4 — Frontend: Memory Panel Enhancements

### Why

The current `MemoryPanel.jsx` renders result cards with a snippet, timestamp, source label, and a Reference button. Three additions make the panel significantly more useful: relevance scores (so the user can judge result quality), source-type filtering (semantic vs episodic), and a delete button on semantic entries. The Reference button and existing search behavior are unchanged.

### Touch points

- `frontend/src/components/MemoryPanel.jsx`:

  **Score display:**
  - For each result card, render the `score` field (from `MemorySearchItem.score`) as a small formatted badge alongside the source label: e.g. `semantic  0.823`
  - If `score` is `null` (episodic results have `score: null`), render `—` in place of the number
  - Score is displayed read-only; no interaction

  **Source filter:**
  - Add a filter bar above the result list with three toggle options: `All` (default), `Semantic`, `Episodic`
  - Filter state is local component state (`useState`)
  - Filtering is client-side on the already-fetched `results` array — no additional API call
  - When filter is `Semantic`, show only results where `source === "semantic"`
  - When filter is `Episodic`, show only results where `source === "episodic"`

  **Delete button (semantic only):**
  - For results where `source === "semantic"` and `metadata.id` is present, render a small `Delete` button on the result card alongside the existing Reference button
  - On click, call `deleteMemoryEntry(result.metadata.id)` (added in T18.5)
  - On success (204 or 200): remove the result from the local `results` state immediately (optimistic removal) and show a brief inline confirmation
  - On error: show an inline error message on that card without affecting other cards
  - Episodic results show no delete button (episodic is append-only by design)

  **No changes** to the search input, handleSearch, Reference button behavior, or panel layout/styling beyond the additions above.

### Acceptance

- Score badge renders correctly for semantic results (numeric) and episodic results (`—`).
- Source filter toggles correctly filter the displayed result list without re-fetching.
- Delete button appears only on semantic results that have `metadata.id`.
- Clicking Delete calls `deleteMemoryEntry` and removes the card on success.

## T18.5 — Frontend: `deleteMemoryEntry()` in `taskClient.js`

### Why

`taskClient.js` has `searchMemory()` but no delete function. The `DELETE /memory/semantic/{id}` endpoint added in T18.1 needs a client binding before T18.4 can call it.

### Touch points

- `frontend/src/api/taskClient.js`:
  - Add:
    ```javascript
    export async function deleteMemoryEntry(entryId) {
      const response = await fetch(
        `${API_BASE_URL}/memory/semantic/${encodeURIComponent(String(entryId))}`,
        { method: 'DELETE' }
      )
      if (!response.ok) {
        const errorText = await response.text()
        throw new Error(`DELETE /memory/semantic failed: ${response.status} ${errorText}`)
      }
      return response.json()
    }
    ```

### Acceptance

- `deleteMemoryEntry(id)` issues `DELETE /memory/semantic/{id}` and returns the parsed JSON response.
- `deleteMemoryEntry(id)` throws on non-OK response with status and body text.
- `npm --prefix frontend run build` — PASS

### Evidence

- `npm --prefix frontend run build` — PASS (covered by T18.4 build)
