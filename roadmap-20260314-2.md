# roadmap-20260314-2.md

## Milestone M16.2: Ollama Host Escalation Provider
**Date:** 2026-03-14
**Status:** Draft ‚Äî Pending Approval

---

### Goal

M16.2 adds a localhost Ollama escalation tier that sits between the container-local GGUF model path and the existing cloud API-key escalation providers. When the container-local model is unavailable and Ollama escalation is enabled, the controller attempts Ollama on the host machine first. Only if Ollama is also unavailable or misconfigured does the existing `decide_escalation()` / cloud path proceed.

This is a meaningful addition to the local-first invariant from `Project.md` ¬ß1 and ¬ß3.3: Ollama on the host is still local execution ‚Äî no data leaves the machine ‚Äî making it a strictly preferable fallback to cloud escalation before any API-key provider is attempted.

**Design constraints:**
- Ollama is kept entirely separate from `SUPPORTED_PROVIDERS`, `ApiKeyRegistry`, and `_ESCALATION_PROVIDER_REGISTRY`. Those structures model keyed external providers; Ollama has no API key and is local-network only. Mixing Ollama into those abstractions would distort their semantics.
- `decide_escalation()` and `escalation_policy.py` are not modified. The Ollama pre-check fires in `_handle_local_model_unavailable()` before `decide_escalation()` is called.
- Redaction is **not** applied to the Ollama path. Ollama is local inference; applying strict PII redaction before a local call removes context without privacy benefit. The original (unredacted) user input is passed to Ollama.
- Ollama connectivity failure is non-terminal: fall through silently to the existing cloud escalation path.
- No authentication, no model discovery, no health-check endpoint, no Ollama API tags listing.

## T16.2.1 ‚Äî Ollama Settings Fields + `.env` / `.env.example` Additions

### Why

Three new settings govern Ollama escalation: a master gate, a base URL for reaching host Ollama from inside the container, and a model name. All three follow the same environment-variable-driven configuration pattern as the existing escalation settings in `settings.py`.

`OLLAMA_BASE_URL` defaults to `http://host.docker.internal:11434` ‚Äî the correct address for reaching a host-side service from inside a Docker container on Docker Desktop (Windows and macOS). On Linux Docker Engine, `host.docker.internal` is not available by default; T16.2.2 adds the `extra_hosts` compose config that makes this alias available on Linux as well. The default value is therefore valid and portable across all platforms once T16.2.2 is applied.

`OLLAMA_MODEL` is required when `ALLOW_OLLAMA_ESCALATION=true`. An empty model name with escalation enabled is a misconfiguration that must be handled gracefully at call time (skip Ollama, fall through), not at settings load time.

`ALLOW_OLLAMA_ESCALATION` and `OLLAMA_MODEL` are hot-applied editable settings. `OLLAMA_BASE_URL` requires a restart (it affects connection initialization at provider call time, but changing it mid-session is confusing; restart semantics are consistent with other URL-type settings).

### Touch points

- `backend/config/settings.py`:
  - Add to `Settings`:
    ```python

    OLLAMA_BASE_URL: str = "http://host.docker.internal:11434"
    OLLAMA_MODEL: str = ""
    ```
  - Extend `SafeConfigProjection` TypedDict with `allow_ollama_escalation: bool`, `ollama_base_url: str`, `ollama_model: str`
  - Extend `get_safe_config_projection()` to include all three fields
  - Add to `EDITABLE_SETTINGS_ENV_KEYS`: `"allow_ollama_escalation": "ALLOW_OLLAMA_ESCALATION"`, `"ollama_model": "OLLAMA_MODEL"` (hot-applied)
  - `OLLAMA_BASE_URL` is **not** added to `EDITABLE_SETTINGS_ENV_KEYS` (restart-required; operator edits `.env` directly, then restarts)
  - Extend `serialize_editable_setting_value()` for `allow_ollama_escalation` (bool ‚Üí `"true"`/`"false"`) and `ollama_model` (string passthrough)
  - Extend `settings_update_restart_semantics()`: `allow_ollama_escalation` and `ollama_model` are hot-applied; no changes needed for `OLLAMA_BASE_URL` (not API-editable)

## T16.2.2 ‚Äî Docker `extra_hosts` for Linux Host-Gateway

### Why

`OLLAMA_BASE_URL` defaults to `http://host.docker.internal:11434`. On Docker Desktop (Windows and macOS), `host.docker.internal` resolves automatically. On Linux Docker Engine it does not ‚Äî the backend container cannot reach the host by that name without an explicit mapping.

Adding `extra_hosts: ["host.docker.internal:host-gateway"]` to the backend service in `docker-compose.yml` instructs Docker Engine on Linux to map `host.docker.internal` to the host's gateway IP, making the default URL work on all platforms. On Docker Desktop this entry is harmless ‚Äî it is already provided by the Docker daemon and the explicit entry is a no-op.

This is a one-line compose change with no behavioral impact on any other service or on non-Ollama functionality.

### Touch points

- `docker-compose.yml` ‚Äî add to the `backend` service:
  ```yaml
  extra_hosts:
    - "host.docker.internal:host-gateway"
  ```

### Acceptance

- `docker compose config` ‚Äî PASS (valid compose schema, no errors)
- Compose file contains `extra_hosts` entry under the `backend` service with exactly `host.docker.internal:host-gateway`.

## T16.2.3 ‚Äî `OllamaEscalationProvider` Implementation

### Why

The provider is a concrete `EscalationProviderBase` subclass that calls the Ollama REST API. Ollama exposes an OpenAI-compatible endpoint at `/v1/chat/completions` and a native endpoint at `/api/generate`. The OpenAI-compatible path is preferred ‚Äî it uses the same request/response shape as `OpenAIEscalationProvider`, keeping the implementation straightforward with `httpx` (already in `requirements.txt`).

**Key behavioral differences from cloud providers:**
- No API key. The provider reads `OLLAMA_BASE_URL` and `OLLAMA_MODEL` from settings at call time. If `OLLAMA_MODEL` is empty, return `(False, "", "ollama_model_not_configured")` immediately ‚Äî no HTTP call.
- `seed` is passed as a top-level parameter on `/v1/chat/completions` if provided.
- Connection errors (`httpx.ConnectError`, `httpx.TimeoutException`, and similar) are caught and returned as `(False, "", str(exc))` ‚Äî the same contract as all other providers. The controller treats any `False` return as a fall-through, not a terminal failure.
- Timeout: 30 seconds to prevent the Ollama path from blocking indefinitely if the host is slow.

### Touch points

- `backend/models/providers/ollama_provider.py` (new file):
  - `OllamaEscalationProvider(EscalationProviderBase)`:
    - `name = "ollama"`
    - `execute(prompt, max_tokens, seed)`:
      1. Read `settings.OLLAMA_BASE_URL` and `settings.OLLAMA_MODEL` via `Settings()` at call time
      2. If `OLLAMA_MODEL` is empty: return `(False, "", "ollama_model_not_configured")`
      3. POST to `{OLLAMA_BASE_URL}/v1/chat/completions` via `httpx.Client(timeout=30.0)` with `model`, `max_tokens`, `messages`, and `seed` (if not None)
      4. On HTTP 200: extract `response.json()["choices"][0]["message"]["content"]`; return `(True, text, "")`
      5. On any exception (network, timeout, non-200, parse error): return `(False, "", str(exc))`
- `backend/models/providers/__init__.py` ‚Äî add `OllamaEscalationProvider` export

### Acceptance

- `OllamaEscalationProvider().name == "ollama"`
- Returns `(False, "", "ollama_model_not_configured")` when `OLLAMA_MODEL` is empty.
- Returns `(True, text, "")` on mocked successful HTTP 200 response.
- Returns `(False, "", <e>)` on mocked connection failure ‚Äî does not raise.
- `Settings()` is called inside `execute()`, not at `__init__` ‚Äî same settings-at-call-time pattern as cloud providers.
- No `OllamaEscalationProvider` added to `_ESCALATION_PROVIDER_REGISTRY`.

## T16.2.4 ‚Äî Ollama Pre-Check Block in Controller

### Why

The Ollama attempt must fire before `decide_escalation()` ‚Äî not as a peer of the cloud providers, but as a preferred local fallback that the policy layer does not need to know about. The insertion point is the top of `_handle_local_model_unavailable()`, before any policy evaluation.

**Execution sequence after this task:**
```
1. container-local model unavailable ‚Üí _handle_local_model_unavailable() called
2. [NEW] if ALLOW_OLLAMA_ESCALATION=true:
       attempt OllamaEscalationProvider().execute()
       on success ‚Üí set context["ollama_status"] = "succeeded", set llm_output, return
       on failure ‚Üí set context["ollama_status"] = "failed" or "not_configured", fall through
3. decide_escalation() evaluated against cloud ESCALATION_PROVIDER setting
4. deny ‚Üí escalation_status = "denied", skip_llm = True
5. allow ‚Üí redact prompt ‚Üí _ESCALATION_PROVIDER_REGISTRY.get(provider) ‚Üí execute()
```

**Context keys added by the Ollama block:**
- `ollama_status`: `"not_attempted"` (escalation disabled), `"succeeded"`, `"failed"`, `"not_configured"` (model empty)
- No changes to existing `escalation_status`, `escalation_code`, `escalation_reason`, or `escalation_decision` keys ‚Äî those belong to the cloud path and are unchanged.

**Redaction:** the original `context["user_input"]` is passed to Ollama without redaction. Ollama is local inference; the same input that would go to the container-local model is appropriate here.

### Touch points

- `backend/controller/controller_service.py`:
  - Import `OllamaEscalationProvider` from `backend.models.providers`
  - At the top of `_handle_local_model_unavailable()`, before `fallback_message` is constructed and before `decide_escalation()` is called, insert:
    ```python
    if bool(settings.ALLOW_OLLAMA_ESCALATION):
        ollama_provider = OllamaEscalationProvider()
        ollama_ok, ollama_output, ollama_error = ollama_provider.execute(
            prompt=str(context.get("user_input", "")),
            max_tokens=512,
            seed=self.generation_seed,
        )
        if ollama_ok:
            context["ollama_status"] = "succeeded"
            context["escalation_status"] = "not_attempted"
            context["llm_output"] = str(ollama_output)
            context["skip_llm"] = False
            return
        else:
            context["ollama_status"] = (
                "not_configured" if ollama_error == "ollama_model_not_configured"
                else "failed"
            )
            context["ollama_error"] = str(ollama_error)
            # fall through to cloud escalation path
    else:
        context["ollama_status"] = "not_attempted"
    ```
  - No other changes to `_handle_local_model_unavailable()`, `_ESCALATION_PROVIDER_REGISTRY`, or any other controller method.

### Acceptance

- `ollama_status == "succeeded"` and `llm_output` set from Ollama when Ollama path succeeds; cloud path not attempted.
- `ollama_status == "failed"` and fall-through to cloud path when Ollama returns `(False, ...)`.
- `ollama_status == "not_configured"` and fall-through when `OLLAMA_MODEL` is empty.
- `ollama_status == "not_attempted"` when `ALLOW_OLLAMA_ESCALATION=false`.
- Existing cloud escalation paths (`escalation_status == "denied"`, `"escalated"`, `"failed"`) are unaffected when Ollama is disabled or falls through.
- On Ollama success, `escalation_status` is set to `"not_attempted"` to indicate the cloud path was bypassed.
- Normal task paths where the local model is found are completely unaffected.
