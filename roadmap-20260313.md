## Milestone M16: Escalation Provider Execution Layer
**Date:** 2026-03-13
**Status:** Draft — Pending Approval

---

### Goal

M16 closes G-A: the escalation provider execution layer is absent. Every surrounding layer is complete and verified — settings, `.env` parity, API key registry, policy evaluation, redaction, controller integration, UI provider selection. The only missing piece is concrete `EscalationProviderBase` implementations for each supported provider and their registration in `_ESCALATION_PROVIDER_REGISTRY` at startup.

Currently, when `decide_escalation()` returns `allow=True` (all seven policy gates pass), the controller executes:

```python
provider_impl = _ESCALATION_PROVIDER_REGISTRY.get(provider)
if provider_impl is None:
    context["escalation_status"] = "failed"
    context["escalation_error"] = "escalation_provider_not_registered"
```

This is the terminal failure in all production configurations. The goal of M16 is to replace this dead end with real HTTP dispatch for each provider whose API key is configured.

`Project.md` §3.3 and §6.3 require cloud model fallback with policy-bound execution. The M15 control plane is the foundation; M16 is the execution layer that makes it functional.

**Design constraints carried from AGENTS.md and Project.md invariants:**
- Local-first: escalation is attempted only after local model is confirmed unavailable and policy gates pass. No change to this sequencing.
- Deterministic control: the FSM/DAG drives execution; provider selection is deterministic for the same settings state.
- Traceability: every escalation attempt produces a decision artifact in context; successful escalation output is recorded in episodic trace like any other LLM output.
- Policy-bound: `decide_escalation()` output is the sole authority on whether dispatch proceeds. Provider implementations do not re-evaluate policy.
- Privacy: redaction is already applied before the registry lookup in M15. Provider implementations receive a pre-redacted prompt; they must not log, cache, or store the original.

## T16.2 — Concrete Provider Implementations

### Why

`EscalationProviderBase` defines the contract: `execute(prompt, max_tokens, seed) -> tuple[bool, str, str]`. Each provider implementation wraps its SDK's completion call in this interface. The implementation layer is the only place SDK-specific behavior lives; all callers remain provider-agnostic.

**Design rules for all implementations:**
- Read the API key via `ApiKeyRegistry().get_api_key(self.name)` at call time, not at instantiation — avoids stale key references if `.env` is reloaded.
- On any SDK exception, return `(False, "", str(exc))` — never raise; the controller handles failure.
- `seed` parameter: pass to providers that support deterministic sampling (OpenAI `seed`, Anthropic `seed` parameter if supported); ignore silently for providers that do not.
- `max_tokens`: pass through directly. Callers are responsible for sizing.
- No logging of prompt content, completion content, or API key values inside implementations.
- No retry logic — single attempt only; retry policy is a future concern.

### Touch points

- `backend/models/providers/` (new directory):
  - `__init__.py` — exports all four concrete classes
  - `anthropic_provider.py` — `AnthropicEscalationProvider(EscalationProviderBase)`:
    - `name = "anthropic"`
    - Uses `anthropic.Anthropic(api_key=...)` client
    - Calls `client.messages.create(model="claude-opus-4-6", max_tokens=max_tokens, messages=[{"role": "user", "content": prompt}])`; extracts `response.content[0].text`
    - Returns `(True, text, "")` on success; `(False, "", str(exc))` on any exception
  - `openai_provider.py` — `OpenAIEscalationProvider(EscalationProviderBase)`:
    - `name = "openai"`
    - Uses `openai.OpenAI(api_key=...)` client
    - Calls `client.chat.completions.create(model="gpt-4o", max_tokens=max_tokens, messages=[{"role": "user", "content": prompt}], seed=seed)`; extracts `response.choices[0].message.content`
    - Returns `(True, text, "")` on success; `(False, "", str(exc))` on any exception
  - `gemini_provider.py` — `GeminiEscalationProvider(EscalationProviderBase)`:
    - `name = "gemini"`
    - Uses `google.generativeai.configure(api_key=...)` + `google.generativeai.GenerativeModel("gemini-2.0-flash")`
    - Calls `model.generate_content(prompt)`; extracts `response.text`
    - Returns `(True, text, "")` on success; `(False, "", str(exc))` on any exception
  - `grok_provider.py` — `GrokEscalationProvider(EscalationProviderBase)`:
    - `name = "grok"`
    - Uses `openai.OpenAI(api_key=..., base_url="https://api.x.ai/v1")` client (Grok uses OpenAI-compatible API)
    - Calls `client.chat.completions.create(model="grok-3", max_tokens=max_tokens, messages=[{"role": "user", "content": prompt}])`; extracts `response.choices[0].message.content`
    - Returns `(True, text, "")` on success; `(False, "", str(exc))` on any exception

### Acceptance

- Each implementation subclasses `EscalationProviderBase` and implements `execute()` with the correct return type signature.
- No implementation raises an exception for any input — all errors are captured and returned as `(False, "", error_str)`.
- `name` attribute on each matches the corresponding key in `SUPPORTED_PROVIDERS` exactly.
- API key is read via `ApiKeyRegistry().get_api_key(self.name)` inside `execute()`, not at `__init__`.
- All four classes exported from `backend/models/providers/__init__.py`.

## T16.3 — Register Providers in `_ESCALATION_PROVIDER_REGISTRY`

### Why

`_ESCALATION_PROVIDER_REGISTRY` is declared as an empty module-level dict in `controller_service.py`. It is never populated in production. T16.3 populates it at module load time with all four concrete providers from T16.2. Registration is unconditional — a provider is registered regardless of whether its key is currently configured. The policy gate `PROVIDER_KEY_MISSING` already handles the case where a registered provider has no key; the registry is a dispatch table, not a key-validity filter.

### Touch points

- `backend/controller/controller_service.py`:
  - Add imports for all four concrete providers from `backend.models.providers`
  - Populate `_ESCALATION_PROVIDER_REGISTRY` at module level with all four entries:
    ```python
    from backend.models.providers import (
        AnthropicEscalationProvider,
        GeminiEscalationProvider,
        GrokEscalationProvider,
        OpenAIEscalationProvider,
    )

    _ESCALATION_PROVIDER_REGISTRY: dict[str, EscalationProviderBase] = {
        "anthropic": AnthropicEscalationProvider(),
        "gemini": GeminiEscalationProvider(),
        "grok": GrokEscalationProvider(),
        "openai": OpenAIEscalationProvider(),
    }
    ```
  - No other changes to `_handle_local_model_unavailable()` — the existing registry lookup, status setting, and error handling are correct as implemented in M15.
  - No changes to any other controller method, workflow node, or DAG executor.

### Acceptance

- `_ESCALATION_PROVIDER_REGISTRY` contains all four provider entries at module load time.
- `_ESCALATION_PROVIDER_REGISTRY.get("anthropic")` returns an `AnthropicEscalationProvider` instance (and likewise for the other three).
- No provider instantiation raises an exception at import/module-load time regardless of `.env` state.
- `escalation_status` remains `"not_attempted"` on all task paths where the local model is found successfully — no behavioral change for the normal path.
- `escalation_status` becomes `"escalated"` (not `"failed"`) when policy gates pass and the provider is now registered.

## T16.4 — Tests

### Why

All four provider implementations must be testable without live network calls. The controller's production dispatch path — which was previously only reachable with a monkeypatched registry — is now the default; the registry is populated at module load. Tests must verify the full dispatch path for each provider, confirm correct failure handling, and confirm the normal local-model path remains unaffected.

All provider tests use `respx` (already in `requirements.txt`) or `unittest.mock.patch` to intercept HTTP calls. No live API calls in any test.

### Touch points

- `tests/unit/test_escalation_providers.py` (new):
  - One test class per provider (`TestAnthropicProvider`, `TestOpenAIProvider`, `TestGeminiProvider`, `TestGrokProvider`), each containing:
    - `test_execute_returns_text_on_success` — mock SDK response; assert `(True, "<text>", "")` returned
    - `test_execute_returns_false_on_sdk_exception` — mock SDK to raise; assert `(False, "", <non-empty str>)` returned; assert no exception propagates
    - `test_name_matches_supported_providers` — assert `provider.name in SUPPORTED_PROVIDERS`
    - `test_api_key_read_at_call_time` — assert `ApiKeyRegistry().get_api_key()` is called inside `execute()`, not at `__init__` (via mock or monkeypatch of `ApiKeyRegistry`)
  - `test_all_providers_registered` — import `_ESCALATION_PROVIDER_REGISTRY` from `controller_service`; assert all four provider names present as keys
- `tests/unit/test_controller_service_integration.py` — additions:
  - `test_controller_escalation_dispatches_to_registered_anthropic_provider` — monkeypatch `AnthropicEscalationProvider.execute` to return `(True, "anthropic_output", "")`; configure settings for escalation; assert `escalation_status == "escalated"` and `llm_output == "anthropic_output"`
  - `test_controller_escalation_dispatches_to_registered_openai_provider` — same pattern for OpenAI
  - `test_controller_escalation_provider_failure_sets_failed_status` — monkeypatch `execute` to return `(False, "", "api_error")`; assert `escalation_status == "failed"`, `escalation_error` set
  - `test_controller_normal_path_unaffected_by_registry_population` — provide a valid local model; assert `escalation_status == "not_attempted"` and normal `llm_output` produced

### Acceptance

- All new tests pass.
- Test count increases above `337` baseline.
- No existing tests broken.
- No live HTTP calls in any test; all SDK calls mocked.
- All four provider `execute()` success and failure paths have isolated unit coverage.
- Controller dispatch path to a registered real provider has integration coverage.
