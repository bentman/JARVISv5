## T13.1 — Complete `GET /memory/search` Endpoint

### Why
`MemorySearchItem` and `MemorySearchResponse` schemas are already defined in `schemas.py`
and imported in `main.py`. `test_api_memory_search.py` exists in `tests/unit/`. The endpoint
itself was never added to `main.py`. The work is structurally half-done.

`EpisodicMemory.search_decisions()` and `MemoryManager.retrieve_knowledge()` are both
implemented and tested at the memory layer. This task adds the missing API surface that
connects those capabilities to callers.

**Touch points:** `backend/api/main.py`, `tests/unit/test_api_memory_search.py`
(update to cover the now-live endpoint).

### Acceptance
- `GET /memory/search?q=<query>` returns results from both semantic and episodic memory.
- Each result includes source type, content, and timestamp.
- Empty or missing `q` returns a clear validation error, not a 500.
- Zero-results returns an empty list, not an error.
- Existing endpoints unaffected.
- Unit tests cover query, empty-query, and zero-results cases and pass.

### Evidence
- `backend/.venv/Scripts/python -m pytest tests/unit/test_api_memory_search.py -q` — PASS excerpt
- `backend/.venv/Scripts/python scripts/validate_backend.py --scope unit` — PASS excerpt

## T13.2 — Research Intent Routing

### Why
`RouterNode` is four lines. `Project.md` §5.1 defines three intent paths: `chat`, `code`,
`research`. §2 lists "search, research synthesis" as primary daily-use paths. Research intent
— which should route toward the `tool_call`/search pipeline — does not exist.

The `tool_call` node, search providers, and provider ladder are all operational. The workflow
graph compiler (`compile_plan_to_workflow_graph`) always emits the same linear graph
regardless of intent. This task adds a deterministic research classifier to `RouterNode` and
connects `intent = research` to the search execution path.

Classification must remain rule-based (keyword/heuristic). Deterministic control is a
`Project.md` core invariant — no LLM-driven routing decisions.

**Touch points:** `backend/workflow/nodes/router_node.py`,
`backend/workflow/plan_compiler.py` (or `controller_service.py` — wherever the graph is
selected based on intent), `tests/unit/test_nodes.py` or equivalent router test.

### Acceptance
- `intent = research` is produced deterministically for inputs expressing search or lookup intent.
- `intent = code` and `intent = chat` behaviour is unchanged.
- Research-intent tasks route toward the `tool_call`/search execution path in the workflow graph.
- Same input always produces the same intent classification.
- All three intent paths covered by unit tests.

### Evidence
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS excerpt covering router
  intent classification and research workflow routing.

## T13.3 — Validator Quality Depth

### Why
`ValidatorNode` is three lines: `is_valid = bool(str(output).strip())`. Any non-empty string
passes. `Project.md` §3.1 defines validation gates as "fail-closed policies with explicit
error artifacts." §5.1 states the validator "verifies output quality and format compliance."

A rule-based quality validator should at minimum detect and fail on: outputs that are pure
whitespace or punctuation after stripping, outputs that echo the prompt structure verbatim
(leaked system prompt fragments, role labels), and outputs containing known inference
artifacts (stop tokens, format leakage). Failure reason must be written into context as an
explicit, machine-readable artifact — not a silent boolean.

This remains rule-based. No LLM judge. Deterministic control invariant preserved.

**Touch points:** `backend/workflow/nodes/validator_node.py`,
`tests/unit/test_nodes.py` or a dedicated validator test file.

### Acceptance
- Validator fails on known bad output patterns with explicit failure reason in context.
- Valid assistant output passes without regression.
- Failure reason is accessible in context for downstream telemetry use.
- Empty/whitespace output still fails (existing behaviour preserved).
- Unit tests cover each distinct fail category and representative pass cases.

### Evidence
- `backend/.venv/Scripts/python -m pytest tests/unit/ -q` — PASS excerpt

## T13.4 — Memory Search Panel (Frontend)

### Why
`Project.md` §3.5 requires "memory/search controls" in the web client. §11.2 names it as
an interface element. §2 lists "knowledge recall" as a daily-use path. The `GET /memory/search`
endpoint (T13.1) provides the backend surface. This task adds the frontend panel that makes
it accessible to the user.

Pattern: follow the existing `SettingsPanel.jsx` / `WorkflowVisualizer.jsx` panel pattern —
slide-in panel, accessible from the header, polled or query-driven. No new architectural
patterns needed.

Minimum viable surface: a query input, a results list showing source type / content /
timestamp, and a way to reference a result in the current chat (copy-to-input or inject).

**Touch points:** `frontend/src/components/MemoryPanel.jsx` (new),
`frontend/src/api/taskClient.js` (add `getMemorySearch` client method),
`frontend/src/App.jsx` (panel toggle, consistent with existing settings/workflow toggles).

### Acceptance
- Memory panel is accessible from the UI header, consistent with existing panel pattern.
- User can submit a query and receive results from backend memory search.
- Results display source type, content snippet, and timestamp.
- Panel open/close does not affect chat state.
- `npm --prefix frontend run build` passes clean.

### Evidence
- `npm --prefix frontend run build` — PASS excerpt
